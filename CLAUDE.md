# CLAUDE.md

The shape of this repository can be discovered by reading its files. What cannot be discovered that way, and therefore belongs here, are the operational disciplines that govern how changes are made and the reasoning behind them.

## The marketplace index must mirror the filesystem

The `source` field in `marketplace.json` must exactly match the directory name under `plugins/` that it points to. This is not a stylistic preference. When the index and the filesystem disagree, marketplace resolution does not fail gracefully on the offending entry; it fails for the entire marketplace. The blast radius of a single mismatched name is every plugin in the repository, not just the one that was renamed. Whenever a plugin is added or renamed, treat the index update as part of the same act, not as a follow-up. A rename reaches further than the index; the full set of artifacts that move together, and how to carry users' enabled-plugins setting forward, live in `docs/renaming-plugins.md`.

## Permissions describe what a skill needs, not what it might want

A skill's `allowed-tools` should list only the tools it actually uses. The temptation to add Bash "just in case" trades a small convenience now for a much larger cost later: when something goes wrong, the set of things that could have gone wrong is bounded by what was permitted. Every tool added beyond what is needed widens that bound and makes post-incident reasoning harder. Least privilege here is not a quality measure; it is a ceiling on how badly a failure can unfold.

## A skill that is not a user action does not belong in the slash menu

A skill is exposed to two audiences at once: Claude loads it when relevant, and the user runs it from the slash menu. When a skill's body is knowledge Claude reads rather than an action a user runs, the menu is the wrong place for it — the entry invites the user to execute something with nothing to perform, and dilutes the menu against the commands that are real actions. The test is whether invoking the skill triggers anything the user means to trigger; if not, keep it available to Claude but remove it from the menu. This is least privilege applied to a skill's surface rather than its permissions: expose to each audience only what that audience can use.

## A change is not done until its dependent artifacts are done

A plugin change is complete only when the code, the `version` in `plugin.json`, and the `CHANGELOG.md` entry all move together. Committing the code first and leaving the version and changelog for later breaks the correspondence between the release tag and what the tag actually contains, within the very commit range that produced it. Tags and releases are generated by CI, so the moment these three artifacts are out of sync locally, the recorded history becomes permanently wrong. The judgment of which version component to bump lives in `docs/versioning.md`.

## The changelog speaks to users, not to maintainers

A changelog entry should describe what changes for someone who installs the plugin and uses it, not what changed inside the source tree. Internal restructurings, renumbered principles, prose rewrites, and the reasoning behind a refactor are all visible in the code and the commit history; repeating them in the changelog buries the entries that actually matter to a reader deciding whether to upgrade. When a single change has several internal facets, collapse them into the one line that names the externally observable difference. The collapse is justified only when the facets are internal to one observable difference; when a change produces two differences a user notices separately, each earns its own line. The deciding question is never how many lines look tidy but whether a user would register one change or two by using the plugin, so collapsing two distinct user-visible differences to save a line buries one of them as surely as listing an internal facet adds noise. The version number already lives in the section heading, so the body should not restate it. The test for any line is whether a user of the plugin would recognize the described change by using the plugin; if not, the line belongs in the commit message, not here.

A change that produces no user-observable difference at all does not earn a changelog entry by virtue of having required a version bump. The instinct is to translate an internal-only change into user-facing language so that it can be listed, but this produces a line that passes the recognition test only because it has been abstracted to the point of meaninglessness — and a meaningless entry crowds out the entries a reader is actually scanning for. When the honest answer to "would a user notice this?" is no, the change is recorded in the commit history and the version bump alone, and the changelog stays silent on it. Silence is the correct entry for an internal-only change, not a softened paraphrase.

## Unreleased changes go under [Unreleased]

New changelog entries belong under the existing `## [Unreleased]` heading. Do not invent a new `## [X.Y.Z] - YYYY-MM-DD` section when adding an entry. The version number and the release date are decided when a release is cut, not when a change is written; CI reads the latest version section to produce the tag and the GitHub release, so any version-and-date heading committed locally pre-commits a decision that belongs to the release step. The date in particular will almost certainly be wrong by the time the change actually ships, and once it is in the recorded history the discrepancy is permanent. Whoever cuts the release promotes `[Unreleased]` to a version heading; the contributor's job is to leave the entries where the release step can find them.

## Stay inside the scope you were asked to change

When working on a plugin, do not silently start modifying a different plugin because something in it looks improvable. Noticing that a neighbor has the same bug, or could benefit from the same refactor, is valuable information, but acting on it without surfacing it first changes the scope of the work the user agreed to. The user reviews, approves, and reasons about a change at the granularity they requested. Quietly expanding that granularity makes the change harder to review, harder to revert, and harder to attribute when something later breaks. If a related issue is worth addressing, raise it as a separate proposal and let the user decide whether to widen the scope.

## A plugin's principles are written as principles, not as a list of cases

Some plugins govern their own behavior through a constitution — a file of numbered principles that the plugin reads to decide what to do in situations the author never enumerated. The value of such a file is entirely in its generality: a principle stated as a value, with the reasoning that makes it true, applies to cases that did not exist when it was written. A principle pinned to the specific symptom that prompted it does not. When a concrete incident motivates a new principle, the discipline is to find the general value the incident is an instance of and state that value, so the principle reaches the next, unforeseen instance rather than only the one already seen. Naming the triggering symptom in the principle's body inverts this: a future reader treats the named case as the definition and overlooks the principle when the situation does not match. Let the body carry the value; if the originating case must be recorded at all, the changelog and the commit history are where it belongs, not the principle.

## Recurring mistakes belong in this document

When the same kind of mistake or the same drift in behavior happens more than once, the right response is not to fix it again more carefully. It is to recognize that an unwritten assumption is being relied on, and to propose adding it here as an explicit principle. Anything that lives only in memory or intuition will be lost the moment the context shifts. Only what is written down can be referenced in a future session and checked against actual behavior.
