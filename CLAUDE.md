# CLAUDE.md

The shape of this repository can be discovered by reading its files. What cannot be discovered that way, and therefore belongs here, are the operational disciplines that govern how changes are made and the reasoning behind them.

## The marketplace index must mirror the filesystem

The `source` field in `marketplace.json` must exactly match the directory name under `plugins/` that it points to. This is not a stylistic preference. When the index and the filesystem disagree, marketplace resolution does not fail gracefully on the offending entry; it fails for the entire marketplace. The blast radius of a single mismatched name is every plugin in the repository, not just the one that was renamed. Whenever a plugin is added or renamed, treat the index update as part of the same act, not as a follow-up.

## Permissions describe what a skill needs, not what it might want

A skill's `allowed-tools` should list only the tools it actually uses. The temptation to add Bash "just in case" trades a small convenience now for a much larger cost later: when something goes wrong, the set of things that could have gone wrong is bounded by what was permitted. Every tool added beyond what is needed widens that bound and makes post-incident reasoning harder. Least privilege here is not a quality measure; it is a ceiling on how badly a failure can unfold.

## A change is not done until its dependent artifacts are done

A plugin change is complete only when the code, the `version` in `plugin.json`, and the `CHANGELOG.md` entry all move together. Committing the code first and leaving the version and changelog for later breaks the correspondence between the release tag and what the tag actually contains, within the very commit range that produced it. Tags and releases are generated by CI, so the moment these three artifacts are out of sync locally, the recorded history becomes permanently wrong. The judgment of which version component to bump lives in `docs/versioning.md`.

## The changelog speaks to users, not to maintainers

A changelog entry should describe what changes for someone who installs the plugin and uses it, not what changed inside the source tree. Internal restructurings, renumbered principles, prose rewrites, and the reasoning behind a refactor are all visible in the code and the commit history; repeating them in the changelog buries the entries that actually matter to a reader deciding whether to upgrade. When a single change has several internal facets, collapse them into the one line that names the externally observable difference. The version number already lives in the section heading, so the body should not restate it. The test for any line is whether a user of the plugin would recognize the described change by using the plugin; if not, the line belongs in the commit message, not here.

A change that produces no user-observable difference at all does not earn a changelog entry by virtue of having required a version bump. The instinct is to translate an internal-only change into user-facing language so that it can be listed, but this produces a line that passes the recognition test only because it has been abstracted to the point of meaninglessness — and a meaningless entry crowds out the entries a reader is actually scanning for. When the honest answer to "would a user notice this?" is no, the change is recorded in the commit history and the version bump alone, and the changelog stays silent on it. Silence is the correct entry for an internal-only change, not a softened paraphrase.

## Stay inside the scope you were asked to change

When working on a plugin, do not silently start modifying a different plugin because something in it looks improvable. Noticing that a neighbor has the same bug, or could benefit from the same refactor, is valuable information, but acting on it without surfacing it first changes the scope of the work the user agreed to. The user reviews, approves, and reasons about a change at the granularity they requested. Quietly expanding that granularity makes the change harder to review, harder to revert, and harder to attribute when something later breaks. If a related issue is worth addressing, raise it as a separate proposal and let the user decide whether to widen the scope.

## Recurring mistakes belong in this document

When the same kind of mistake or the same drift in behavior happens more than once, the right response is not to fix it again more carefully. It is to recognize that an unwritten assumption is being relied on, and to propose adding it here as an explicit principle. Anything that lives only in memory or intuition will be lost the moment the context shifts. Only what is written down can be referenced in a future session and checked against actual behavior.
