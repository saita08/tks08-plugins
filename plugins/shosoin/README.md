# shosoin

> 「その仕様、CLAUDE.md に書いといて」
> 「README にも書いておきますね」
> 「docs/ にも同じ説明があった気がする」
> 「三箇所とも、微妙に違うことが書いてありますが」
> 「……どれが本当？」

三冊の本に三通りの真実が書いてある書庫を、人は書庫と呼びません。呼ぶとしたら、物置です。`shosoin` は、その物置を、千三百年物が残る蔵に変えるプラグインです。

名前は正倉院から取りました。奈良に実在する宝庫で、八世紀の紙も布もガラスも、朽ちずに今も残っています。秘訣は魔法ではなく、運用です。扉は勅封で、開けるには許しが要る。年に一度は宝物を取り出して風を通し、状態を検める。残すことを構造と習慣にした蔵だけが、千年後も蔵であり続けます。

## これは何？

プロジェクトの書かれた知識を、それぞれ性質の違う四つの棚と一枚の案内板に配架し、収めた宝物がいまも真実かを検める蔵です。

- **CLAUDE.md** は蔵の憲章です。価値観と判断基準だけを書きます。コードベースを明日ゼロから書き直しても生き残る文だけが、ここに住む資格を持ちます
- **docs/** は現在の目録です。「今この瞬間に何があるか」だけを書きます。構成、手順、データ契約。読むのはコードを書く前で、書いた後の言い訳のためではありません
- **adr/** は決定の書庫です。「なぜそう決めたか、何を捨てたか」を時系列で綴じます。一度採用された記録は書き換えず、判断が変わったら新しい一枚で上書きではなく置き換えます。すべての決定が不変の記録として時系列に刻まれていく様は、世界の全記憶が記されるというアカシックレコードに少し似ています。ただしこちらは実在して、grep が効きます。書庫はルート直下に建ちます。決定の記録は現状説明の押し入れに仕舞う物ではないからです
- **references/** は標本室です。スプレッドシートのサンプル行、捕獲したAPIレスポンス。散文にせず、加工せず、コードからも読み込まない生データの棚です
- **README.md** は門前の案内板です。初めて来た人に、ここが何の蔵で、どの棚に何があるかを示します

配架の原則はただ一つ、**一つの事実に、住所は一つ**。同じ内容が二つの棚にあると、現実が動いたときにどちらか一方だけが更新され、読者はもうどちらも信じられなくなります。腐った文書は、信頼されているぶん、無い文書より悪いのです。

## 特徴

- **配架には必ず証拠を添えます**: 「このファイルのこの節は、書き直し後も生き残る価値観ではなく今日の手順なので、docs/ 行きです」。ファイル名の雰囲気で宝物を動かす蔵守は、蔵守ではなく引っ越し業者です
- **曝涼が中身の嘘を暴きます**: 正倉院が年に一度宝物に風を通して検めるように、点検は各文書の主張をディスク上の現実と突き合わせ、「ページは X と言うが、リポジトリは Y を示す」の形で報告します。採用後にこっそり書き換えられた ADR は、git の履歴が覚えています
- **扉は勅封です**: 監査も点検も、まず報告を差し出し、あなたが頷いた分だけ蔵に手を入れます。移動は `git mv` で履歴ごと運び、リンクと索引も同じ一手で追随させます。リンク切れを残す移動は、移動ではなく遺失です
- **綺麗な蔵には綺麗と言います**: 違反ゼロの監査報告も、全書合格の点検報告も、空振りではなく成果です。仕事を見せるために問題を発明したりはしません
- **空の棚は作りません**: 生データの無いプロジェクトに references/ は建てません。空の標本室は家具ではなく、ただの埃です

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/shosoin:audit` | 全文書を四層に写像し、配架違反を証拠つきで報告。承認された配架案だけを実行する |
| Command | `/shosoin:inspect` | 曝涼。各文書の主張を現実と突き合わせ、古びた記述・切れたリンク・書き換えられた記録を報告する |
| Command | `/shosoin:setup` | 文書構造の無いプロジェクトに、四層の棚と案内板を実際の観察に基づいて建立する |
| Skill | `four-track-contract` | 四層知識契約の教義・配架手順・点検手順・雛形。「この情報どこに書く?」に随時答える |

## 使い方

散らかった蔵には監査を。

```
/shosoin:audit
```

中身が信じられなくなったら曝涼を。

```
/shosoin:inspect
```

更地には建立を。

```
/shosoin:setup
```

会話の途中で「これって docs/ と README どっちに書くべき?」と聞くだけでも、スキルが教義から答えます。コマンドの儀式は要りません。

## setup が最初にやる、少し行儀のいいこと

建立のとき、蔵は最初の二枚の決定記録を自分で書きます。「ADR を使うという決定」と「四層契約を採用するという決定」です。日付は建立した当日、却下した代替案も正直に記します。「決定を記録せよ」と説く構造が自分の誕生の記録を持っていなかったら、その蔵は落成の日から自己矛盾で軋むからです。

## よくある質問

**Q: 全部 CLAUDE.md に書いてはだめですか？**
A: だめではありません。ただし価値観と現状と判断理由を一冊に綴じると、現状が変わるたびに理由が消えるか、理由を守って現状の記述が腐るか、必ずどちらかが起きます。三年もののCLAUDE.mdの中で、今も真実の段落がどれか当てるゲームをしたことがある人は、この分割の理由をすでに体で知っています。

**Q: adr/ がルートにあるのはなぜですか。docs/adr/ のほうが片付いて見えます。**
A: 片付いて見えるのは、決定の記録を現状説明の子分として扱っているからです。「今どうなっているか」と「なぜそうしたか」は親子ではなく、対等な別トラックです。書庫を書斎の押し入れに入れる蔵はありません。

**Q: 配架が正しければ、中身も正しいですよね？**
A: 別の問題です。全部の宝物が正しい棚にあって、その全部の目録が三年前の現実を語っている蔵は普通に存在します。だから棚の監査と中身の曝涼は別のコマンドです。棚は audit、中身は inspect が受け持ちます。

**Q: 勝手にファイルを動かしませんか？**
A: 動かしません。扉は勅封です。蔵は配架案と点検報告を差し出し、あなたが頷いた分だけ手を入れます。二つの棚に等しく収まりうる宝物は、勝手に分類せず、あなたに尋ねます。宝物の意味を決めるのは蔵守の仕事ではないからです。

**Q: 一度整理したら、もう散らからないですよね？**
A: 散らかります。蔵は、使われている限り散らかり続けます。だから曝涼は年中行事なのです。大きな変更のあとに、また audit と inspect を呼んでください。

**Q: 私のプロジェクトは小さくて、四層も要らない気がします。**
A: 正しい感覚です。この契約は「知識があるならここに住む」を定めるもので、棚を埋めるために知識を製造しろとは言いません。二枚のファイルで足りる蔵には、二枚のファイルと、それがどこにあるかの案内板だけで十分です。

**Q: なぜ正倉院ですか？**
A: 一度も焼けたことがなく、千三百年間、収めた物を残し続けている蔵だからです。秘訣は勅封と曝涼、つまり「勝手に開けない」と「定期的に検める」の二つでした。このプラグインがやるのは、その二つの真似です。名前ごと縁起を担ぎました。

## できないこと・正直な制約

- **点検は照合であって、審査ではありません。** 文章の巧拙や設計の良し悪しは判定しません。間違った推論を正確に記録した ADR は、点検の目には健全な記録です
- **照合できない主張は「照合できない」と報告します。** 実機やリモートの状態を語る文書の真偽は、リポジトリの中からは確かめられません。references/ の標本と実機のずれも、コード側の期待と食い違ったときに間接的に気づけるだけです
- **古びた文書の書き直しは提案までです。** 死んだリンクや改名されたパスは承認のうえ直しますが、消えたアーキテクチャを語るページの再執筆は、書ける知識を持つ人の仕事として報告します。発見を閉じるために中身を発明したら、古い真実が新しい虚構に変わるだけだからです
- **分類は最終的に人間の判断です。** 価値観と現状の境界には、本当に曖昧な宝物が存在します。蔵は基準と推薦を示しますが、票は持ちません
- **静的サイト生成ツールが所有する docs/ には手を出しません。** mkdocs.yml や docusaurus.config が棚の構造を決めている蔵では、発見は報告に留め、動かすかどうかはあなたが決めます

宝物が千年残るのは、蔵が頑丈だからではありません。開けるたびに検めて、必ず元の場所へ返す人がいたからです。
