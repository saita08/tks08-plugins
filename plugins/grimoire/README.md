# grimoire

> 「エージェントを会話で率いるのはもう飽きた」
> 「毎回同じ指示を口移しで、同じ順番で、同じように待つ」
> 「これ、書き留めておけば二度と唱えなくて済むのでは」
> 「……という魔法を、また今日も口伝で唱えている」

魔導士がいつまでも呪文を暗唱していては三流です。強い魔導士は呪文を書き留め、必要なときに開いて詠唱し、改良し、自分の呪文を書き足していく。`grimoire` は、あなたのために一冊の魔導書を差し出します。実戦を生き延びたWorkflowスクリプトを呪文として収め、詠唱・改変・追記ができる書です。

## これは何？

エージェントの統率には二つの段階があります。会話で率いる段階と、プログラムで統率する段階。前者は柔軟ですが、毎回同じことを口で説明し直す。後者は Claude Code の Workflow、決定論的なスクリプトでエージェントの軍団を指揮する仕組みです。繰り返せて、途中から再開でき、後から監査できる。会話では手に入らない三つの性質です。

問題は、Workflow を一から書くのが面倒なことです。スキーマのトップレベルを配列にすると API が黙って空を返す罠、バリアを張るべきか流すべきかの判断、予算に応じた並列数の調整。知っていれば一分、知らなければ半日溶ける類の知識が要る。`grimoire` はその知識を教義として同梱し、さらに書けば動く呪文を四本、最初から棚に並べておきます。

## 同梱の四つの呪文

魔導書を開くと、すぐ詠唱できる呪文が四本収まっています。説明文そのものが、いつ唱えていつ唱えてはいけないかを語ります。

`exhaustive-audit` は、対象が枯れるまで監査を繰り返す呪文です。一度の監査は見つかるものを見つけて終わりますが、この呪文は既知の指摘を次の周回に渡し、前の周が見逃した分だけを狩る。一周目が潰した明白なバグの陰に隠れていた二番手が、二周目でようやく姿を見せる。敵対的な検証者が指摘を上げ、懐疑家が一つずつ真偽を確かめ、新しい指摘が出なくなった周で止まります。

`great-migration` は、一つの機械的変更を大量の対象に施す呪文です。対象を列挙し、各自を隔離された worktree で並列に変換する。worktree の隔離があるから、隣り合うファイルをいじる二体が互いの書きかけを踏むことがない。難しい一件が易しい残りを堰き止めないよう、検証は対象ごとに走ります。ただし `sed` 一発で済む変換に一ファイルずつエージェントを差し向けるのは、竜を呼んで蠅を叩く所業です。呪文の説明はそれもちゃんと諫めます。

`truth-panel` は、一つの主張を複数の審査員が別々のレンズで敵対的に検証し、評決を下す呪文です。正しさ・安全性・保守性・性能、それぞれの審査員が他には見えない破綻を見る。評決は多数決ではなく重み付けです。一枚のレンズが本物の破れを見つけたら、それは残り全員が承認した主張を沈められる。

`deep-survey` は、一体では抱えきれない広い問いに答える呪文です。問いを facet に割って並列に走査し、有望なものだけを深読みし、最後に一本の報告へ縫い合わせる。走査は広く安く、深読みは選んで高く、統合は各 facet の調査員が誰も立てなかった位置から矛盾と接続を見つける。

## 三つの詠唱

このプラグインは三つのコマンドで魔導書を扱います。

`/grimoire:cast` は詠唱です。呪文名と対象を渡すと、呪文を読み、対象に合わせて args とプロンプトを書き換え、Workflow として走らせ、結果を報告する。呪文の「いつ使うか」節は関門でもあり、対象が明らかに合わなければ、唱える代わりに正直に断ります。

`/grimoire:inscribe` は追記です。あなたがこのセッションで組み立てた Workflow を、呪文の形式に蒸留してあなたの個人魔導書に書き足す。蒸留とは、一つの対象に貼り付いていた具体値を args へ持ち上げ、構造だけを残すこと。動いたスクリプトをそのまま保存するのは、実行可能なログであって呪文ではありません。

`/grimoire:index` は目録です。同梱と個人、両方の全呪文を一覧する。同名の呪文があれば、詠唱時に勝つのは個人の版です。

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/grimoire:cast` | 呪文を対象に合わせて適応させ、Workflowとして詠唱する |
| Command | `/grimoire:inscribe` | 直前に組んだWorkflowを呪文の形式に蒸留し、個人魔導書に書き足す |
| Command | `/grimoire:index` | 同梱と個人、全呪文の目録を一覧する |
| Skill | `spellcraft` | Workflow authoring の技法と、実戦済み呪文の収蔵庫 |

## 使い方

```
/grimoire:index
```

まず目録を開いて、棚に何があるかを見ます。それから対象を選んで詠唱します。

```
/grimoire:cast exhaustive-audit src/auth
```

呪文は雛形であって固定の呪句ではないので、`cast` は対象に合わせて args とプロンプトを書き換えてから走らせます。走り終わると、呪文の目的に応じた報告が返る。監査なら指摘の一覧、移行なら対象ごとの成否、審査なら評決、調査なら統合報告。呪文が worktree にファイルを書いた場合、あなたが承認するまで本物のツリーは無傷です。

自分で組んだ Workflow を書き足したくなったら、`/grimoire:inscribe`。個人魔導書 `~/.claude/grimoire/` に落ち、同梱の書を上書きせず拡張します。

## 前提

- Claude Code の Workflow が使えること。この魔導書の呪文はすべて Workflow スクリプトです。
- 個人魔導書は `~/.claude/grimoire/` に置かれます。`inscribe` が初回に作るので、あらかじめ用意する必要はありません。

## よくある質問

**Q: スラッシュコマンドから Workflow を呼んで大丈夫なの？**
A: 大丈夫です。ユーザーがコマンドで明示的に指示した Workflow の呼び出しは、正当な opt-in であることが公式仕様で保証されています。あなたが目録を開き、対象を選び、詠唱する。全部あなたの意思で始まります。

**Q: スキーマのトップレベルを配列にしたら空が返るって本当？**
A: 本当です。API が 400 で撥ね、エラーはスクリプトに浮かんでこないまま、エージェントは何も返しません。Workflow が黙って壊れる最も多い原因です。だから同梱呪文はすべて配列を `{ items: [...] }` の形にくるんであり、技法の教義もこの罠を真っ先に警告します。知らなければ半日、知っていれば一行です。

**Q: 同梱の呪文を書き換えたい。**
A: 書き換えるのではなく、同じ名前で個人魔導書に書けば、詠唱時にはあなたの版が勝ちます。同梱の書は読み取り専用のまま、あなたの改変はその上に重なる。原本を汚さずに自分の魔導書を育てる、それがこの書の拡張の作法です。

**Q: 呪文を唱え間違えたら暴走する？**
A: しません。Workflow は決定論的で、途中で止めても resume で続きから再開できます。乱数も現在時刻も呪文には使えない決まりで、唱えるたびに同じ盤面が再現されます。制御を失う呪文は、そもそも書けないようになっています。

書は開かれるのを待っています。暗唱はもう終わりにしましょう。
