# pluginize

> 「これ、前にもやったな」
> 「同じ手順、今週3回目だ」
> 「でもプラグインにするの、面倒なんだよな」
> 「……で、今日もやらない」

そうして見送られ続ける「プラグイン化」を、向こうから声をかけてくれるのが、このプラグインです。

## これは何？

作業をしていると、再利用できる構造がしばしば生まれます。繰り返しの手順、固まった段取り、誰も書き残していないコツ。本来ならプラグインにして次から楽をすべきものです。でも「気づく」のも「作る」のも腰が重く、たいてい流れていきます。

`pluginize` は、その**気づきと起動の摩擦**を引き受けます。あなたが作業を一区切りつけるたび、裏で静かに「いま再利用できる構造が生まれなかったか」を見ています。何も無ければ黙っています。候補が見つかったときだけ、「これ、プラグインにしませんか?」と一度だけ声をかけます。

そして、あなたが「やる」と言ったら、**プラグインを作る重い作業はこのセッションの外**でやります。今の会話は汚れません。

## 三つの層

このプラグインは三つの層でできていて、それぞれに一つの規律が乗っています。

### 観測 — 広く見て、割り込まない

Stop hook が、あなたのターンが終わるたびに裏で軽く `claude -p` を走らせ、再利用できる構造の候補を探します。判断軸は三つ。**再現性**（同じ作業がまた来る）、**手続きの結晶度**（段取りの順番が固まっている）、**暗黙知**（非自明なコツに頼った）。どれか一つでも当てはまれば候補です。

検出は広めに拾います。見逃した候補は二度と戻ってきませんが、拾いすぎた候補はあなたが一言「いらない」と言えば済むからです。だから絞るのは検出側ではなく、あなたの採否の側に置いています。そして候補が無ければ、完全に沈黙します。声をかけるのは例外で、沈黙が常態です。

### 提案・承認 — 効果を添えて、勝手に動かない

候補が見つかったら、ただ挙げるだけでなく「プラグインにするとこういう得がある」を添えて差し出します。あなたが承認するまで、生成は一切始まりません。

この承認には二つの意味があります。一つは**あなたの意図の確認**。もう一つは、このプラグイン自身の**無害性の証明**です。勝手に動かないことを、ドキュメントではなく振る舞いで示します。だから過去の承認は引き継ぎません。提案のたびに、承認は取り直します。

### 隔離生成 — 重さを、ここに持ち込まない

承認されたら、`claude --bg` で隔離されたセッションを起動し、そこで [`/plugin-dev:create-plugin`](https://code.claude.com/docs/en/plugins) を走らせます。プラグインの作り方を知っているのはそのワークフローで、`pluginize` はそれに任せます。

メインセッションがやるのは、候補をまとめて投げるところまで。あとは別セッションの仕事です。生成セッションは agent view から覗いて対話できるので、プラグインを作り込む会話はそちらで進めてください。`claude --bg` のファイル編集は worktree に隔離され、生成物はあなたのローカルに着地します。

## 使い方

インストールすれば、観測層は自動で働きます。あなたは普通に作業を続けるだけです。

候補が見つかると、ターンの区切りで一度だけ提案が差し込まれます。乗り気なら `/pluginize:propose-plugin` を実行してください。候補が承認ゲートを通り、生成セッションが立ち上がります。乗り気でなければ、無視してそのまま続けて構いません。

hook の提案を待たず、自分から「これプラグインにできる?」と `/pluginize:propose-plugin` を呼んでも構いません。

## 生成物はローカルに着地する。公開は勧めるだけ

生成されたプラグインは、まずあなたのローカルプロジェクトに着地します。汎用的に見えても、それは変わりません。ローカルが検証の場であり、公開は不可逆で、「汎用かどうか」の判定は外れることもあるからです。

もし汎用的に見えたら、生成ワークフローは最後に「公開を検討してみては」と**勧めます**。実行はしません。マーケットプレースがあるか、公開したいか、どうやるかは、あなたの領分です。

## 構成

```
pluginize/
├── .claude-plugin/
│   └── plugin.json
├── hooks/
│   ├── hooks.json          # Stop hook（asyncRewake で裏で観測、候補時のみ rewake）
│   └── detect.sh           # transcript を claude -p に渡し3軸で候補判定。再帰ガード付き
├── skills/
│   └── propose-plugin/
│       ├── SKILL.md         # 承認 → claude --bg で生成セッションを起動
│       └── references/
│           ├── detection-axes.md    # 何をプラグイン化すべきか
│           ├── plugin-value.md      # プラグイン化すると何が得られるか
│           ├── constitution.md      # 各層を貫く価値と判断の枠組み
│           └── generation-prompt.md # 生成セッションに渡すプロンプトの雛形
└── README.md
```

## 前提

- `claude` CLI がパスにあること。観測（`claude -p`）も生成（`claude --bg`）もこれを使います。API キーは不要で、Claude Code の認証をそのまま流用します。
- `/plugin-dev:create-plugin` を提供するプラグイン（`plugin-dev`）が利用できること。生成は完全にこのワークフローに委譲します。

## FAQ

**毎ターン裏で `claude -p` が走るの? メインのコンテキストは汚れない?**
観測は `asyncRewake` で非同期に走り、メインをブロックしません。候補が無ければ何も出力せず沈黙するので、メインのコンテキストには何も入りません。候補があるときだけ、一行の補足として差し込まれます。

**ユーザースコープでインストールしたら、自分の `settings.json` が上書きされる?**
されません。プラグイン同梱の hooks は、あなたの hooks とメモリ上でマージされて並列実行されます。既存の設定はそのままです。

**`claude -p` を hook から呼んだら、それ自身の Stop hook で無限ループしない?**
`detect.sh` は再帰ガードを持っています。`claude -p` を呼ぶときに環境変数を立て、子セッションの hook はそれを見て即座に抜けます。
