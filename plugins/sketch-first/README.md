# sketch-first

> 「ダッシュボードを作って」
> 「はい、こんな感じでしょうか」（案A）
> 「うーん、違う」
> 「では、こちらは」（案B）
> 「それも違う」
> 「……こちらは」（案C）
> 「全部違う。えっとね、私が欲しいのはね……」（ここから長文の説明が始まる）

このやり取り、順番が逆です。長文の説明は、案を3つ却下した後ではなく、案を出す前に聞くべきだったのです。`sketch-first` は、その順番をひっくり返します。

## これは何？

ユーザーが抽象的なゴールを持っているとき、Claude は親切心から先に具体案を出そうとします。ところがその案は、Claude の頭の中にある「たぶんこういうものだろう」という推測から生まれています。推測がずれていれば案は却下され、却下された案が続くと、最後にユーザーが自分の頭の中の像をまるごと長文で説明し直す羽目になる。案を出したことが、かえって遠回りを生むのです。

`sketch-first` は、案を出す前に手を止めます。そして具体化のかわりに、短い構造化インタビューでユーザーの頭の中にある像を先に引き出します。引き出した像は「スケッチ」――1枚の要件像――にまとめ、ユーザーの「これで合ってる」を取ってから、初めて具体案に進みます。そのときには、もう当てずっぽうではありません。二人で見ている的に向かって撃つのです。

## プランモードと何が違う？

Claude Code には組み込みのプランモードがあります。混同しやすいので、ここをはっきりさせておきます。

- **プランモード**は「どう作るか」の計画です。手順、ファイル、順序を詰める。作るものが何かは、もう決まっている前提で動きます。
- **`sketch-first`** は「そもそも何を作るか」の合意です。作る対象そのものの像を、ユーザーの言葉で、実装の話が出てくる前に固める。

この二つは競合しません。むしろ順番に使います。まず `sketch-first` で「何を作るか」を合意し、次にプランモードで「どう作るか」を詰める。先にスケッチを取っておくと、プランが初めてレビューする価値を持ちます。的外れなものを作る計画は、丁寧であればあるほどたちが悪い――その丁寧さが、間違った的を「決まったこと」に見せてしまうからです。

## 何を聞くのか

インタビューには五つの軸があります。どれも「これを聞かずに案を出すと却下される」という理由から選ばれています。

- **誰のためか。** その前に座って使うのは誰で、来たときに何を知っているのか。作った本人向けと初見の訪問者向けは、同じ言葉でも別の製品です。
- **何が変わったら成功か。** 「何を持っているか」ではなく「うまくいったとき世界の何が変わるか」。機能の一覧は設計であって、ゴールではありません。
- **何に似ていて、何に似てはいけないか。** 「Xみたいな」は、どんな抽象的な説明よりも速く像を引き出します。そして「Yみたいにはしないで」は、しばしばそれ以上に雄弁です。踏み込んではいけない領域をまるごと消してくれるからです。
- **絶対に譲れない点。** それを外したら他がどんなに良くても全部台無しになる、一つか二つの壁。
- **どうでもいい点。** 明示的な「気にしない」。これはほぼ必ず聞き忘れられますが、制約と同じくらい価値があります。ユーザーがそもそも判定する気のなかった選択で悩むのをやめられるからです。

どの軸も、ユーザーの言葉で――製品として見て判断できる形で――聞きます。フレームワークやデータモデルを知らないと答えられない質問は、軸を間違えています。そして、実際に曖昧さのある軸だけを聞きます。五つを機械的に全部聞くのは、守ろうとしている「ユーザーの注意」を自分で浪費する行為です。

## 使い方

作るもの・決めることが抽象的で、いきなり案を出すと外しそうなとき、`/sketch-first` を呼びます。引数に対象を渡してもいいし(`/sketch-first オンボーディング画面`)、直前に頼んだことをそのまま対象にしても構いません。

Claude はまず軸に沿ってインタビューし、答えを1枚のスケッチにまとめ、「これで頭の中の像と合ってる?」と確認します。合っていなければ直します。直しはインタビューが機能している証拠であって、遠回りではありません。スケッチが確定して初めて、具体的な作業が始まります。

スキルも同梱しています。あなたが `/sketch-first` を呼ばなくても、抽象的で作り直しコストの高い依頼をしたとき、Claude が「先にスケッチを取りませんか」と控えめに提案することがあります。乗らなければ無視して構いません。

## スケッチはどこに残るか

確定したスケッチは、プロジェクトの `.claude/sketch-first.local.md` に書き出されます。セッションをまたいで残るので、後の作業がいつでも「そもそも何を作る約束だったか」に立ち返れます。中身は人間が読める Markdown で、軸の答えは YAML frontmatter に、スケッチ本文は散文で入ります。すでにファイルがあれば、上書きせず、今ある内容を見せてから直すか聞きます。

## できないこと・正直な制約

- **実装計画は作りません。** それはプランモードの領分です。`sketch-first` が決めるのは「何を」であって「どう」ではありません。
- **コードは書きません。** このコマンドが残す成果物はスケッチ1枚だけです。
- **五つの軸を全部は聞きません。** 曖昧さが実在する軸だけを聞きます。全部聞くのが丁寧なのではなく、要らない質問をしないのが丁寧です。
- **明確な依頼や使い捨ての試作には向きません。** 「`--verbose` フラグを足して」のように読みが一つしかない依頼や、数秒で作り直せるものに対しては、インタビューは純粋な overhead です。曖昧で、かつ作り直しが高くつくとき――その二つが揃ったときだけ、このプラグインは元を取ります。
- **AskUserQuestion を使います。** インタビューはこのツールで対話的に行われるので、非対話的な実行環境では十分に働きません。
