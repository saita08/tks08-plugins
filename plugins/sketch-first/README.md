# sketch-first

> 「ダッシュボードを作って」
> 「はい、こんな感じでしょうか」（案A）
> 「うーん、違う」
> 「では、こちらは」（案B）
> 「それも違う」
> 「……こちらは」（案C）
> 「全部違う。えっとね、私が欲しいのはね……」（ここから長文の説明が始まる）

このやり取り、順番が逆です。長文の説明は、案を3つ却下した後ではなく、案を出す前に聞くべきだったのです。`sketch-first` は、絵を描き始める前にまず取材する似顔絵師です。目の前の人がどんな顔を思い描いているかを聞き出してから、鉛筆を握ります。

## これは何？

ユーザーが抽象的なゴールを持っているとき、Claude は親切心から先に具体案を出そうとします。ところがその案は、Claude の頭の中にある「たぶんこういうものだろう」という推測から生まれた似顔絵です。似ていなければ破り捨てられ、破られた紙が積み上がると、最後にユーザーが自分の顔を鏡ごと持ってくる羽目になる。早く描き始めたことが、かえって遠回りを生むのです。

`sketch-first` は、鉛筆を置いて取材から始めます。短い構造化インタビューで、ユーザーの頭の中にある像を先に聞き出す。聞き出した像は「スケッチ」――1枚の要件像――にまとめ、ユーザーの「これで合ってる」を取ってから、初めて具体案に進みます。そのときにはもう当てずっぽうではありません。二人で同じ顔を見ながら描くのです。

## プランモードと何が違う？

Claude Code には組み込みのプランモードがあります。混同しやすいので、ここをはっきりさせておきます。プランモードは「どう描くか」の話です。画材、手順、仕上げの順序。何を描くかはもう決まっている前提で動きます。`sketch-first` は「そもそも誰の顔を描くか」の合意です。描く対象そのものの像を、ユーザーの言葉で、画材の話が出てくる前に固める。

この二つは競合しません。むしろ順番に使います。まず `sketch-first` で「何を描くか」を合意し、次にプランモードで「どう描くか」を詰める。的外れなものを作る計画は、丁寧であればあるほどたちが悪い。その丁寧さが、間違った顔を「決まったこと」に見せてしまうからです。

## 何を聞くのか

取材には五つの軸があります。どれも「これを聞かずに描くと破られる」という理由から選ばれています。

- **誰のためか。** その前に座って使うのは誰で、来たときに何を知っているのか。作った本人向けと初見の訪問者向けは、同じ言葉でも別の製品です。
- **何が変わったら成功か。** 「何を持っているか」ではなく「うまくいったとき世界の何が変わるか」。機能の一覧は設計であって、ゴールではありません。
- **何に似ていて、何に似てはいけないか。** 「Xみたいな」は、どんな抽象的な説明よりも速く像を引き出します。そして「Yみたいにはしないで」は、しばしばそれ以上に雄弁です。踏み込んではいけない領域をまるごと消してくれるからです。
- **絶対に譲れない点。** それを外したら他がどんなに良くても全部台無しになる、一つか二つの壁。
- **どうでもいい点。** 明示的な「気にしない」。これはほぼ必ず聞き忘れられますが、制約と同じくらい価値があります。ユーザーがそもそも判定する気のなかった選択で悩むのをやめられるからです。

どの軸も、ユーザーの言葉で――製品として見て判断できる形で――聞きます。フレームワークやデータモデルを知らないと答えられない質問は、軸を間違えた取材です。そして、実際に曖昧さのある軸だけを聞きます。五つを機械的に全部聞くのは、守ろうとしている「ユーザーの注意」を自分で浪費する、いちばん間の抜けた取材です。

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/sketch-first` | 案を出す前に構造化インタビューを行い、答えを1枚のスケッチにまとめて確認する |
| Skill | `sketch-interview` | 抽象的で作り直しコストの高い依頼を見抜き、先にスケッチを取るべきか判断する知識ベース |

## 使い方

作るもの・決めることが抽象的で、いきなり案を出すと外しそうなとき、`/sketch-first` を呼びます。引数に対象を渡してもいいし(`/sketch-first オンボーディング画面`)、直前に頼んだことをそのまま対象にしても構いません。

Claude はまず軸に沿って取材し、答えを1枚のスケッチにまとめ、「これで頭の中の像と合ってる?」と確認します。合っていなければ描き直します。描き直しは取材が機能している証拠であって、遠回りではありません。似顔絵師が途中で「もう少し目を細く?」と聞くのと同じです。スケッチが確定して初めて、具体的な作業が始まります。

`sketch-interview` スキルも同梱しています。あなたが `/sketch-first` を呼ばなくても、抽象的で作り直しコストの高い依頼をしたとき、Claude が「先にスケッチを取りませんか」と控えめに提案することがあります。乗らなければ無視して構いません。似顔絵師は無理に鉛筆を握らせたりはしません。

## スケッチはどこに残るか

確定したスケッチは、プロジェクトの `.claude/sketch-first.local.md` に書き出されます。セッションをまたいで残るので、後の作業がいつでも「そもそも何を描く約束だったか」に立ち返れます。中身は人間が読める Markdown で、軸の答えは YAML frontmatter に、スケッチ本文は散文で入ります。すでにファイルがあれば、上書きせず、今ある一枚を見せてから描き直すか聞きます。前の似顔絵を勝手に破ったりはしません。

## よくある質問

**Q: 五つの軸、毎回全部聞かれるんですか。**
A: 聞きません。似顔絵師が客に「あなたの目は二つですか」と確かめないのと同じです。曖昧さが実在する軸だけを聞きます。全部聞くのが丁寧なのではなく、要らない質問をしないのが丁寧です。

**Q: `--verbose` フラグを足すだけの依頼でも呼ぶべき?**
A: やめておきましょう。読みが一つしかない依頼や、数秒で作り直せるものに取材を挟むのは、純粋な overhead です。曖昧で、かつ作り直しが高くつくとき――その二つが揃ったときだけ、このプラグインは画材代の元を取ります。

**Q: このコマンド、コードや実装計画は書いてくれる?**
A: 書きません。似顔絵師は絵しか描かないし、それも下絵一枚だけです。「どう作るか」はプランモードの領分で、コードを書くのはその後の工程。ここが残す成果物はスケッチ1枚、それだけです。

**Q: 非対話の環境でも動く?**
A: 十分には働きません。取材は AskUserQuestion で対面式に行うので、客が席にいない画室では、似顔絵師はただ鉛筆を持って待つことになります。

似顔絵師は、描く前に必ず一度、あなたの顔を見ます。
