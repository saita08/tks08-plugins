# mission-control

> 「セッション終わったらSlackに通知するようにしてるんだ」
> 「へえ、なんて来るの?」
> 「『タスクが完了しました』」
> 「……で、何が完了したの?」
> 「それは、通知を見てもわからない」

管制室の無線が「機体が着陸しました」とだけ告げて、どの機体がどこに降りたかは黙っている。それが世の完了通知です。鳴ったことはわかる。中身は、結局アプリを開き直すまでわからない。

そしてもう一つ。リポジトリを何個も抱えている管制官は、毎朝それを一機ずつ呼び出して点呼を取ります。PRが滑走路で止まっていないか、CIが赤ランプを灯していないか、自分待ちのレビューが管制塔に溜まっていないか。タブを開いては閉じ、開いては閉じ。コーヒーが冷める頃、ようやく空域の全体像が見えてきます。

`mission-control` は、この二つを引き受ける管制室です。完了通知には「何が着陸したか」を乗せ、朝の点呼は一枚のブリーフィングに束ねます。

## これは何？

二つの部品でできています。片方はセッションが終わるたびに黙って無線を飛ばすフック、もう片方はあなたが管制塔から呼び出したときだけ動くコマンドです。

## 特徴

- **中身の乗った完了通知**: セッションが着陸すると、transcript から最後のアシスタントメッセージの要点を拾ってSlackに流します。固定文言ではなく、実際にそのセッションで何が起きたかが無線に乗ります。「終わった」ではなく「何が終わったか」が届く、それだけのことに世界は長らく苦しんできました
- **黙秘する既定**: Webhookを設定していなければ、フックは一文字も喋らず即座に引き返します。このマーケットプレースは一度に大量のフックを回すので、設定していない人にとってこのフックは無線が切れているのと区別がつきません。それが正しい姿です
- **朝の点呼を一枚に**: `/mission-control:brief` を呼ぶと、リポジトリ群を横断して `gh` を回し、滞留PR・赤いCI・あなた待ちのレビューを一枚のブリーフィングに束ねます。全部きれいなら「今朝は静かです」と一言。管制室は静かな朝を報告できる管制室です
- **見るだけ、撃たない**: ブリーフィングは読み取り専用。マージも、クローズも、コメントも、再実行もしません。管制室は状況を読み上げる係で、操縦桿を握るのはあなたです

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Hook | `Stop` → `notify.sh` | セッション完了時、最後のアシスタントメッセージの要点をSlackへ。webhook未設定なら沈黙 |
| Command | `/mission-control:brief` | リポジトリ群を横断し、滞留PR・赤いCI・レビュー待ちを一枚のブリーフィングに |

## 使い方

### 完了通知

環境変数 `SLACK_WEBHOOK_URL` を設定すれば、それだけでセッション完了ごとに無線が飛びます。プロジェクト単位で設定・切り替えしたい場合は `.claude/mission-control.local.md` に webhook を書きます（下記「設定」を参照）。環境変数があればそちらが優先で、無ければ `.local.md` を見に行きます。

設定しなければ、無線は飛びません。それが正しい既定です。

### ブリーフィング

```
/mission-control:brief
```

これだけで、対象リポジトリを横断したブリーフィングが一枚返ってきます。追加で見たい空域があれば、その場限りで足せます:

```
/mission-control:brief owner/another-repo
```

どのリポジトリを巡回するかは設定ファイルに列挙しておけますし、無ければカレントリポジトリだけを見ます。引数で名指ししたリポジトリは、設定ファイルの分に上乗せされます。名指ししていない空域まで勝手に偵察に出ることはありません。それは遅いし、何より人を驚かせます。

### 毎朝、勝手に点呼を取る

ブリーフィングはスケジューラと組ませると本領を発揮します。Claude Code 組み込みの `/schedule`（あるいは `claude` を叩く任意の cron）を `/mission-control:brief` に向けておけば、あなたが枕から頭を上げる前に管制室が点呼を済ませています。このプラグインはスケジューラ自体を同梱しません。ブリーフィングを用意するところまでが管制室の仕事で、いつ回すかはあなたの時計に委ねます。

## 設定

プロジェクト単位の設定は `.claude/mission-control.local.md` に置きます。YAML frontmatter に対象リポジトリと、任意で Slack webhook を書きます。

```markdown
---
repos:
  - tks08/tks08-plugins
  - tks08/some-service
webhook: https://hooks.slack.com/services/xxx/yyy/zzz
---

このファイルの本文には、人間向けのメモを自由に書けます。
frontmatter の webhook は通知フックとブリーフィングの両方が共有します。
```

- `repos:` は `/mission-control:brief` が横断する対象。省略すればカレントリポジトリのみ。
- `webhook:` は通知フックと、ブリーフィングの投稿が共有します。ブリーフィングの投稿は毎回あなたのyesを取ってからにします。環境変数 `SLACK_WEBHOOK_URL` を設定していれば、通知フックはそちらを優先します。

このファイルは秘密情報（webhook URL）を含みます。バージョン管理に載せないよう、`.gitignore` に `.claude/*.local.md` を足しておくのが安全です。

## 前提

- 通知フックは `curl` と `python3` を使います。どちらも多くの環境に既に着任しています。Slack webhook は「Incoming Webhooks」で発行したURLを想定しています。
- `/mission-control:brief` は `gh`（GitHub CLI）がパスにあり、認証済みであることを前提とします。無い・未認証のときは、その旨を告げて止まります。管制室が勝手にインストールも認証もすることはありません。

## よくある質問

**Q: 既に自前のStop→Slackフックを持っています。**
A: それなら二重に無線が鳴ります。このプラグインの通知フックと、あなたが既に仕込んでいる通知が両方鳴る、ということです。どちらかに寄せてください。要点抽出つきの通知が欲しければ自前のを外す。既存ので十分ならこのフックは有効化しないか、Webhookを設定しないでおけば黙秘します。

**Q: 通知に乗るのはセッション全体の要約ですか?**
A: いいえ、「最後のアシスタントメッセージの要点」です。結論を最後のメッセージにまとめる普段の書き方とは相性がいい一方、結論が途中に埋もれたセッションでは要点を取りこぼすことがあります。管制室は最後の交信を読み上げるのであって、フライト全体を録音し直すわけではありません。

**Q: ブリーフィングに映らないものはありますか?**
A: あります。ブリーフィングは `gh` が返すものの上に成り立っています。権限のないリポジトリや、`gh` から見えないCIは映りません。レーダーに映らない機体は、報告にも映らない。管制室はレーダーの外までは見ません。

**Q: Slackの通知は凝った見た目にできますか?**
A: できません。素朴なテキスト一本です。凝ったブロックもスレッド化もしません。管制室は速報を出す係で、清書はしません。それでも「何が終わったか」は、ちゃんと乗っています。
