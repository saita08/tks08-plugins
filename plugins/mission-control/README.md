# mission-control

> 「セッション終わったらSlackに通知するようにしてるんだ」
> 「へえ、なんて来るの?」
> 「『タスクが完了しました』」
> 「……で、何が完了したの?」
> 「それは、通知を見てもわからない」

完了通知あるあるです。鳴ったのはわかる。でも「何が」終わったのかは、結局アプリを開き直さないとわからない。通知が運んでいるのは「終わった」という事実だけで、肝心の中身は置いてけぼりです。

そしてもう一つ。リポジトリを何個も抱えている人は、毎朝それを一個ずつ巡回します。PRが止まっていないか、CIが赤くなっていないか、自分待ちのレビューが溜まっていないか。タブを開いては閉じ、開いては閉じ。朝のコーヒーが冷める頃、ようやく全体像が見えてきます。

`mission-control` は、この二つを引き受ける管制室です。完了通知には「何が完了したか」を乗せ、朝の巡回は一枚のブリーフィングにまとめます。

## これは何？

二つの部品でできています。片方はセッションが終わるたびに黙って働くフック、もう片方はあなたが呼んだときだけ動くコマンドです。

### 完了通知に中身を乗せる — Stop hook

セッションが一区切りつくと、`mission-control` は transcript から**最後のアシスタントメッセージの要点**を拾い、Slackに投げます。固定文言ではありません。実際にそのセッションで何が起きたか――先頭の数行に、`result:` や `まとめ:` のような結論めいた行があればそれも添えて――が通知に乗ります。「終わった」ではなく「何が終わったか」が届きます。

ただしこれは**設定があるときだけ**動きます。Webhookが設定されていなければ、フックは一文字も出力せず即座に引き返します。設定していない人にとって、このフックはいないのと同じです。外の世界(Slack)に何かを送るというのは取り消しのきかない行為なので、明示的に設定された opt-in のときだけ動く、というのが固い約束です。

### 朝の巡回を一枚にまとめる — `/mission-control:brief`

`/mission-control:brief` を呼ぶと、あなたのリポジトリ群を横断して `gh` を回し、**滞留しているPR・CIが赤いところ・あなた待ちのレビュー**を一枚のブリーフィングにまとめます。どのリポジトリを見るかは設定ファイルに列挙しておけますし、無ければカレントリポジトリだけを見ます。引数で `owner/repo` を足せば、その場限りで対象を広げられます。

読み取り専用です。マージも、クローズも、コメントも、再実行もしません。見て、まとめて、渡すだけ。何かを変えるのはあなたの仕事です。

## 使い方

### 完了通知

環境変数 `SLACK_WEBHOOK_URL` を設定すれば、それだけでセッション完了ごとに通知が飛ぶようになります。プロジェクト単位で設定したい・切り替えたい場合は、`.claude/mission-control.local.md` に webhook を書きます(下記「設定」を参照)。

設定しなければ、通知は飛びません。それが正しい既定です。

### ブリーフィング

```
/mission-control:brief
```

これだけで、対象リポジトリを横断したブリーフィングが1枚返ってきます。全部きれいなら「今朝は静かです」と一言。追加で見たいリポジトリがあれば:

```
/mission-control:brief owner/another-repo
```

### 毎朝、勝手に回す

ブリーフィングは、スケジューラと組み合わせると本領を発揮します。Claude Code 組み込みの `/schedule`(あるいは `claude` を叩く任意の cron)を `/mission-control:brief` に向けておけば、毎朝あなたが何もしなくても管制室が巡回を済ませてくれます。このプラグインはスケジューラ自体は同梱しません。ブリーフィングを用意するところまでが仕事で、いつ回すかはあなたのスケジューラに委ねます。

## 設定

プロジェクト単位の設定は `.claude/mission-control.local.md` に置きます。YAML frontmatter に対象リポジトリと(任意で)Slack webhook を書きます。

```markdown
---
repos:
  - tks08/tks08-plugins
  - tks08/some-service
webhook: https://hooks.slack.com/services/xxx/yyy/zzz
---

このファイルの本文には、人間向けのメモを自由に書けます。
（frontmatter の webhook は通知フックとブリーフィングの両方が共有します）
```

- `repos:` は `/mission-control:brief` が横断する対象。省略すればカレントリポジトリのみ。
- `webhook:` は通知フックと、ブリーフィングの投稿(こちらは毎回確認を取ります)が共有します。環境変数 `SLACK_WEBHOOK_URL` を設定していれば、通知フックはそちらを優先します。

このファイルは秘密情報(webhook URL)を含みます。バージョン管理に載せないよう、`.gitignore` に `.claude/*.local.md` を足しておくのが安全です。

## 前提

- 通知フックは `curl` と `python3` を使います(どちらも多くの環境に既にあります)。Slack webhook は「Incoming Webhooks」で発行したURLを想定しています。
- `/mission-control:brief` は `gh`(GitHub CLI)がパスにあり、認証済みであることを前提とします。無い・未認証のときは、その旨を告げて止まります。勝手にインストールも認証もしません。

## できないこと・正直な制約

- **既に自前の Stop→Slack フックを持っているなら、二重通知になります。** このプラグインの通知フックと、あなたが既に仕込んでいる通知が両方鳴ります。どちらかに寄せてください。要点抽出つきの通知が欲しければ自前のを外し、既存ので十分ならこのフックは有効化しないか、Webhookを設定しないでおけば黙ります。
- 通知に乗るのは「最後のアシスタントメッセージの要点」であって、セッション全体の要約ではありません。結論を最後のメッセージにまとめる普段の書き方とは相性が良い一方、結論が途中に埋もれるセッションでは要点を取りこぼすことがあります。
- ブリーフィングは `gh` が返すものの上に成り立っています。権限のないリポジトリや、`gh` から見えないCIは映りません。見えないものは、報告にも映らないということです。
- Slackの通知フォーマットは素朴なテキスト一本です。凝ったブロックやスレッド化はしません。管制室は速報を出す係で、清書はしません。
