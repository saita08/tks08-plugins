# new-toys

> 「Claude Code、また新機能出たらしいよ」
> 「へえ、何ができるようになったの?」
> 「リリースノートは読んだ。ちゃんと読んだ」
> 「で、あなたの環境のどこで使うの?」
> 「……それは、書いてなかった」

偵察報告には二種類あります。「敵陣に新しい装備が届いたのを確認」で終わるものと、「その装備は我が方のここに刺さる」まで書いてあるもの。リリースノートは前者です。「新しくXができるようになりました」までは読める。でも「そのXが、自分の設定のどこに刺さるのか」は、リリースノートが翻訳してくれることはありません。新機能はいつも、他人事の顔をして前線にやってきます。

`new-toys` は、後者の報告を上げてくる斥候です。公式のリリースノートまで単身で偵察に出て、未読分を見つけ、それをあなたの実際の設定と導入プラグインに引き当てて「これ、あなたのここで使えます」と持ち帰ります。

## これは何？

コマンドが一つ。フックはありません。斥候は呼ばれたときだけ出撃します。`/new-toys:scout` を呼ぶと、四つの任務を順にこなします。

## 特徴

- **単身偵察**: Claude Code 公式リポジトリの `CHANGELOG.md` まで取りに行きます。地図の主な目的地は `raw.githubusercontent.com` の生ファイル、そこが落ちていれば GitHub のレンダー版へ回り込みます
- **既読照合**: 前回どこまで前進したかを覚えているので、未読分だけを切り出します。バージョンの新旧はCHANGELOG自身の並び順を正とし、semverを手で解釈して間違えるようなことはしません
- **翻訳して持ち帰る**: あなたの `~/.claude/settings.json`（読むだけ）と導入済みプラグインを踏まえ、各新機能に「あなたへの適用案」を添えます。置き換えられそうな自前フック、締められそうな権限、噛み合いそうなプラグイン。抽象的な機能紹介ではなく、あなたの陣地の中の具体的な座標を指します。噛み合わないものには「これはあなた向きではない」と正直に言い、報告全体の信頼を守ります
- **手ぶらで帰ることを恐れない**: 未読が無ければ報告は一言、「新しいおもちゃはありません」。斥候は敵影が無いのに何かをでっち上げたりはしません

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/new-toys:scout` | リリースノートを偵察し、未読の新機能をあなたの設定・プラグインへの適用案つきで報告 |

## 使い方

```
/new-toys:scout
```

これだけです。引数はいりません。呼ぶたびに、前回以降の新機能が「あなたへの適用案」つきで返ってきます。

初回は前回記録がないので、履歴を全部ぶちまける代わりに、直近の一〜二バージョン分を報告して、そこを前線の基準点に定めます。二回目以降は差分だけ。斥候は同じ報告を二度上げません。

## 記録の置き場所

既読の基準点は `~/.claude/new-toys/last-seen.json` に置きます。作戦記録を金庫の奥にしまい込まず、見えるところに人間が読める形で置くのがこのマーケットプレースの流儀です。中身はこれだけ:

```json
{ "last_version": "1.2.3", "seen_at": "2026-07-04T09:00:00Z" }
```

もう一度最初から偵察をやり直したくなったら、このファイルを消すか `last_version` を古い値に書き換えれば、その地点から未読として再出撃できます。

## 読むだけ、提案するだけ

斥候が触れるのは `~/.claude/new-toys/last-seen.json`（記録更新）ただ一つです。`settings.json` もプラグインも、読むだけで一切書き換えません。適用案はあくまで提案で、実際に陣地を組み替えるかどうかはあなたが決めます。斥候は情報を持って帰ってくる係で、陣地を作り替えるのは司令官の仕事です。

## 前提

- `~/.claude/settings.json` があると、適用案が具体的になります。無くても新機能の報告自体は動きます。
- ネットワークに出られること。リリースノートの取得に外部アクセスを使います。前線の外に偵察に出るのですから、道が塞がっていれば出られません。

## よくある質問

**Q: 取得先が変わったら?**
A: そのときは斥候が道に迷います。主な目的地は `https://raw.githubusercontent.com/anthropics/claude-code/main/CHANGELOG.md`。Anthropic がこのファイルの場所や書式（`## <version>` 見出し + 箇条書き）を変えると、取得や解析が崩れます。直し方は単純で、`commands/scout.md` の「1. Fetch the release notes」に書かれた取得先URLを新しい場所に書き換えるだけ。壊れたときは黙って古い地図を差し出さず、「取得できなかった」と告げて止まる作りです。基準点も更新しないので、直したあとにもう一度呼べば取りこぼしなくやり直せます。

**Q: 適用案がどれも一般論なんですが。**
A: それは偵察できる範囲が薄いからです。適用案の質は、あなたの `settings.json` とプラグイン構成をどれだけ読めるかに依存します。設定が薄いと、斥候が持ち帰れる具体性も薄くなります。陣地が整っているほど、報告は鋭くなります。

**Q: プラグインやMCPサーバーの更新も追ってくれますか?**
A: 追いません。報告するのは Claude Code 本体の CHANGELOG です。斥候が偵察するのは本隊の動きで、各部隊の細かい補給までは見ていません。

**Q: この新機能、使うべきですか?**
A: そこまでは言いません。「あなたのここで使えます」までは報告しますが、「使うべきです」とは言いません。採否は司令官の領分です。斥候は地形を報告して、進軍の判断は返します。
