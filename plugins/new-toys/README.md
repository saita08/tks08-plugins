# new-toys

> 「Claude Code、また新機能出たらしいよ」
> 「へえ、何ができるようになったの?」
> 「リリースノートは読んだ。ちゃんと読んだ」
> 「で、あなたの環境のどこで使うの?」
> 「……それは、書いてなかった」

リリースノートを読む習慣がある人でも、ここで詰まります。「新しくXができるようになりました」までは読める。でも「そのXが、自分の設定のどこに刺さるのか」までは、リリースノートは翻訳してくれません。新機能はいつも、他人事の顔をしてやってきます。

`new-toys` は、新機能を偵察してくる斥候です。公式のリリースノートを取りに行き、未読分を見つけ、それを**あなたの実際の設定と導入プラグインに引き当てて**「これ、あなたのここで使えますよ」と報告します。

## これは何？

`/new-toys:scout` を呼ぶと、斥候が四つの仕事をします。

1. **偵察** — Claude Code 公式リポジトリの `CHANGELOG.md` を取りに行きます。
2. **既読照合** — 前回どこまで見たかを覚えているので、未読分だけを切り出します。
3. **翻訳** — あなたの `~/.claude/settings.json`(読むだけ)と導入済みプラグインを踏まえ、各新機能に「あなたへの適用案」を添えます。置き換えられそうな自前フック、締められそうな権限、噛み合いそうなプラグイン――抽象的な機能紹介ではなく、あなたの環境の中の具体的な場所を指します。
4. **記録更新** — どこまで見たかを更新します。次に呼んだときは、また未読分だけが出てきます。

未読が無ければ、報告は一言です。「新しいおもちゃはありません」。無理に何かをひねり出したりはしません。

## 使い方

```
/new-toys:scout
```

これだけです。引数はいりません。呼ぶたびに、前回以降の新機能が「あなたへの適用案」つきで返ってきます。

初回は前回記録がないので、履歴を全部ぶちまける代わりに、直近の一〜二バージョン分を報告して、そこを基準点に設定します。二回目以降は差分だけ。斥候は同じ報告を二度しません。

## 記録の置き場所

既読の基準点は `~/.claude/new-toys/last-seen.json` に置きます。見えるところに、人間が読める形で置くのがこのマーケットプレースの思想です。中身はこれだけ:

```json
{ "last_version": "1.2.3", "seen_at": "2026-07-04T09:00:00Z" }
```

もう一度最初から見直したくなったら、このファイルを消すか `last_version` を古い値に書き換えれば、その地点から未読としてやり直せます。

## 読むだけ、提案するだけ

斥候が触るのは `~/.claude/new-toys/last-seen.json`(記録更新)ただ一つです。`settings.json` もプラグインも、読むだけで一切書き換えません。適用案はあくまで提案で、実際に設定をいじるかどうかはあなたが決めます。斥候は情報を持って帰ってくる係で、陣地を作り替えるのはあなたの仕事です。

## 前提

- `~/.claude/settings.json` があると、適用案が具体的になります(無くても新機能の報告自体は動きます)。
- ネットワークに出られること。リリースノートの取得に外部アクセスを使います。

## できないこと・正直な制約

- **取得先が変われば壊れます。** 斥候は `https://raw.githubusercontent.com/anthropics/claude-code/main/CHANGELOG.md` を主な取得先にしています。Anthropic がこのファイルの場所や書式(`## <version>` 見出し + 箇条書き)を変えると、取得や解析が崩れます。そのときの直し方は単純で、`commands/scout.md` の「1. Fetch the release notes」に書かれた取得先URLを、新しい場所に書き換えてください。壊れたら黙って古い情報を出すのではなく、「取得できなかった」と告げて止まる作りにしてあります(基準点も更新しないので、直したあとにもう一度呼べば取りこぼしなくやり直せます)。
- 適用案の質は、あなたの `settings.json` とプラグイン構成をどれだけ読めるかに依存します。設定が薄いと、適用案も一般的な内容にとどまります。
- 報告するのは Claude Code 本体の CHANGELOG です。個々のプラグインやMCPサーバーの更新までは追いません。斥候が偵察するのは本隊の動きです。
- 新機能の良し悪しを判定はしません。「あなたのここで使えます」までは言いますが、「使うべきです」とは言いません。採否はあなたの領分です。
