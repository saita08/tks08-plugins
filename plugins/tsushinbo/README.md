# tsushinbo

> 「CLAUDE.mdをいじったら、なんか良くなった気がする」
> 「気がする、というと」
> 「気がする、以上のことは、何も測っていない」
> 「では、成績表を発行しましょう」

「より良い」とは比較です。比較には、変更前と変更後の観測が要ります。ところがほとんどの人は、設定をいじって「良くなった気がする」で学期を終えます。`tsushinbo` は、あなたのClaude環境に成績表を発行する個人用の評価基盤です。設定をいじったら試験を受けさせ、点数の推移を見て、その変更が本当に効いたのかを、感想ではなく数字で答えます。

## これは何？

通信簿は、三つの営みでできています。**科目を登録し、試験を実施し、成績表を発行する。** 学校と同じです。ただし生徒はあなたのClaude環境で、成績が上がったり下がったりする原因は、あなたがCLAUDE.mdやプラグインに加えた変更です。

科目とは、再現可能な課題と、観測可能な採点基準と、一言一句固定された採点員のセットです。試験のたびに、予備知識を持たない生徒がその課題を解き、固定の採点員が採点し、点数が環境指紋つきで台帳に刻まれます。学期が進めば、あなたは「あの設定変更のあと、この科目は上がり、あの科目は下がった」を読めるようになります。勘ではなく、記録として。

## 三つの営み

### 登録 — 再現できない課題は科目ではない

`/tsushinbo:register` で、いま会話でやったタスク、あるいはあなたが記述したタスクを科目にします。科目ファイルには三つを書きます。**課題プロンプト**は、この会話を知らない生徒が単独で解けるよう、必要な事実をすべて内側に畳み込んで完結させます。今日の日付や生きたAPIのように、試験のたびに答えが変わる外部状態に依存する課題は、そもそも科目になりません。動く的を撃っても、何も測れないからです。**採点基準**は観測可能な物差しで書きます。「良い感じ」は禁句です。採点員が二度同じに当てられない基準は、基準ではなく好みだからです。

### 試験 — 生徒は予備知識なしで、採点員は固定で

`/tsushinbo:exam` で科目を受験させます。生徒は毎回まっさらな `claude -p` として起動し、採点基準も、採点されていることすら知りません。基準を知った生徒はテストに合わせて答えを作り、その点数は設定の実力ではなく基準の漏れを測ってしまうからです。採点は固定の採点員が、一言一句同じプロンプトで、複数回行い、その中央値を取ります。採点員が試験のたびに変わったら成績は比較不能になり、比較不能な成績は成績表の存在意義そのものを溶かします。点数は環境指紋つきで `grades.jsonl` に追記されます。指紋とは、CLAUDE.mdのハッシュと有効プラグイン一覧のハッシュです。何を変えた時の成績かを、後から特定するためです。

### 発行 — 揺れる物差しは、絶対値ではなく推移で読む

`/tsushinbo:report` で通信簿を発行します。科目別の点数の推移、環境指紋が変わった境目、そして「この設定変更でこの科目は上がり、あの科目は下がった」の対応。採点員の揺れより小さい変動は、動いたことにしません。境目の片側に一度しか受験がなければ、それは結論ではなくヒントだと明記します。三回の受験から一本の線を引いたふりをするより、まだデータが足りないと正直に言うほうが、あなたの役に立つからです。

## 置き場所

台帳と科目は `~/.claude/tsushinbo/` に住みます。科目は `subjects/<科目名>.md` に人間が読めるMarkdownで、成績は `grades.jsonl` に一受験一行で追記されます。台帳の価値は、改竄されていない歴史であることです。だから過去の行は書き換えず、ひたすら追記します。

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/tsushinbo:register` | タスクを科目化する。課題・基準・固定採点員を書き起こす |
| Command | `/tsushinbo:exam` | 科目を受験させ、中央値の点数を環境指紋つきで台帳に刻む |
| Command | `/tsushinbo:report` | 通信簿を発行する。科目別の推移と、設定変更の効果を対応づける |
| Skill | `subject-craft` | 良い科目・観測可能な基準・採点のブレ対策の知識ベース。Claudeが評価基盤を組む前に読む |

## よくある質問

**Q: 採点がLLMなら、点数自体が信用できないのでは？**
A: そのとおりです。物差しは揺れます。通信簿はそれを隠さず、設計の前提にしています。科目を固定して入力を止め、採点員を一言一句凍らせて物差しを固定し、複数回採点して中央値を取り、そして絶対値ではなく推移で読む。点数を絶対の真実として読めば人を惑わせますが、進む方角として読めば、多くの環境がこれまで持ったどんな道具よりも正直な計器になります。

**Q: 科目を1つ登録して1回受験したら、通信簿は出ますか？**
A: 出ますが、それは雑音です。一回の受験は一つのノイズの多い標本であり、意味は受験を重ねた推移に宿ります。科目が少ないうちの通信簿は、ほとんどが雑音です。通信簿はそれを正直に「まだデータが足りない」と言います。誤魔化して結論を捻り出す成績表よりは、そのほうがあなたのためになります。

**Q: 採点員のプロンプトを、あとで良くしたくなったら？**
A: 良くしても構いませんが、それは既存科目の改善ではなく、新しい科目の開始です。採点員が変われば物差しが変わり、前後の点数は別の計器が出したものになって比較できません。採点員を編集することは、成績の歴史を最初からやり直すことだと思ってください。

**Q: 生徒に採点基準を渡してあげたほうが、良い答えを書きませんか？**
A: 書きます。そしてその点数は無意味になります。基準を知った生徒はテストに合わせて答えを作り、あなたが測りたい「設定の実力」ではなく「基準がどれだけ漏れたか」を測ってしまいます。だから生徒には課題だけを渡し、基準は見せません。

**Q: これで良い成績を取れば、私のClaudeは賢いと証明できますか？**
A: できません。証明できるのは「先週のあなたの設定より、今週のあなたの設定が、この科目で上か下か」だけです。通信簿が測るのは絶対的な賢さではなく、あなたの変更の向きです。それで十分なはずです。あなたが本当に知りたいのは、いじった設定が効いたかどうかなのですから。

---

「良くなった気がする」で閉じた学期は、二度と採点できません。通信簿は、あなたの気のせいに、初めて点数をつける道具です。
