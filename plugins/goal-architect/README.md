# goal-architect

> 「`/goal` ってさ、便利そうじゃない？」
> 「条件、なんて書いた？」
> 「『いい感じになるまで』」
> 「それ、永遠に止まらないやつだよ」

そうして生まれたのが、このプラグインです。

## これは何？

このプラグインは、あなたが「いい感じ」としか言えないことを、仮に言葉にしてあげるサービスです。

Claude Code 2.1.139 で追加された [`/goal`](https://code.claude.com/docs/en/goal) は、条件を渡せば毎ターンの後ろで Haiku が「これもう終わった？」を判定して、終わるまで Claude を働かせ続けてくれる、たいへん優秀な機能です。Haiku はやる気に満ちています。問題は、その Haiku に渡す条件文を書ける人類があまりいないことです。

「ちゃんと動く」「だいたい完成」「いい感じ」。日常では通じます。評価モデルから見るとすべて未定義変数です。曖昧に書けば永遠に止まらず、厳しく書けば達成不可能、絶妙に書いたつもりでも「条件は満たしたけど何も終わってない」状態で帰ってきます。三方ハズレです。

このプラグインは、その翻訳作業を引き受けます。あなたの「いい感じ」を、Haiku が首を縦に振れる文章にして返します。代わりに何かを失う気もしますが、`/goal` は止まります。

## 使い方

```
/goal-architect:propose-goal
```

引数なしで起動すると、「今、何を終わらせたいですか？」と聞かれます。漠然としていて構いません。むしろ漠然としているからこのプラグインを呼んだのであって、最初から言語化できているなら自分で `/goal` を打てばいいのです。

```
/goal-architect:propose-goal ログイン機能のリファクタを完了させたい
```

引数で渡すこともできます。出発点としては優秀ですが、そのまま `/goal` に渡すと「リファクタの完了とは……」と Haiku が哲学を始めるので、観察と対話で整形していきます。

最後に `/goal <条件>` の形で文字列が出ます。コピーして実行するのはあなたです。勝手に走らせるのは手抜きじゃなくて、そういう設計です。理由は下の FAQ に書きました。

## なぜ二段階？

このスキルは「何を終わらせたいか」と「条件としてどう書くか」をわざと分けています。

```
段階1: 「こういうことであってる？」「あってる」 ←ここで一旦止まる
段階2: 「じゃあ条件はこうします」 ←ここから初めて /goal の文章にする
```

なぜ分けるかと言うと、人間が思う「終わった」と、評価モデルが見る「条件を満たした」は解像度が違うからです。混ぜると「テストは通ったって書いてある。終了！」「いや、待って、まだ何も直してない」という会話が発生します。発生したあとで「`/goal` 使えなくない？」と言う人を私はたくさん見てきました。境界を保てば起きません。

地味な仕組みですが、ここをサボると `/goal` ごと信用できなくなるので、サボりません。

## 知識は2ファイルに分けてある

スキルの判断材料は `references/` の下に置いてあります。

`goal-spec.md` は `/goal` の事実台帳です。評価モデルは transcript しか見られない、条件文は 4,000 文字まで、`clear` のエイリアスがいくつかある、といった機械の話。仕様が変わったらここを直すと、スキル全体の挙動がついてきます。

`constitution.md` は判断の原則です。良い goal とは何か、どこまで覗くか、ユーザーの言葉と現状が食い違ったらどう振る舞うか。Anthropic の [Claude Constitution](https://github.com/anthropics/claude-constitution) と同じ文体を借りました。ルールに書いてないケースに出くわしたときに自分で判断できる材料を渡す、というのが目的です。

## 何を観察するの？

ユーザーの言葉と現実を突き合わせるために、必要な分だけ覗きます。`CLAUDE.md` や `README` を読み、`git log` で今動いていることを確かめ、失敗中のテストや lint を見て、`gh pr list` で「自覚はあるのに着手していない作業」を発掘し、ユーザーの意図が外部仕様に依存するときだけ `WebFetch` で外に出ます。

ぜんぶを毎回見るわけではありません。意図を確認するのに足りる分だけです。覗きすぎは時間の無駄で、何より気持ち悪いので。

## 構成

```
goal-architect/
├── .claude-plugin/
│   └── plugin.json
├── skills/
│   └── propose-goal/
│       ├── SKILL.md                  手順
│       └── references/
│           ├── goal-spec.md          /goal の仕様
│           └── constitution.md       判断原則
└── README.md
```

軽い構成です。`/goal` 本体に乗っかっているので、hook も agent も MCP も持っていません。「えっ、これだけ？」と言われそうですが、これだけです。必要になったら足します。

## 前提

- Claude Code 2.1.139 以降（`/goal` が使えるバージョン）
- ワークスペースで `/goal` の前提条件を満たしていること（trust dialog 承認済み、`disableAllHooks` が無効）

満たしていないと、頑張って条件を作ったあとに「ところでそれ、どこで実行するんですか？」という静かな問いが残ります。

## FAQ

**Q: なぜ `/goal` を自動で起動しないの？**
A: 自動で起動して、走り出してから「思ってたのと違う」となった場合、止めて戻す責任が誰にあるか曖昧になります。コピペの一手間は、そのための儀式です。あと、`/goal-architect:propose-goal` が暴走しても、あなたの財布は痛みません。

**Q: 引数なしで起動すると何が起きるの？**
A: 「何を終わらせたいですか？」と聞かれます。「分かんない」と答えても怒られません。むしろ歓迎されます。なぜなら、その状態こそ需要のド真ん中だからです。

**Q: 既存の `/goal` の置き換え？**
A: いいえ。`/goal` は組み込みのまま使い続けます。このプラグインは `/goal` に渡す条件文の代筆屋です。あなたが書いたことにしていいです。署名欄はありません。

**Q: 提示された条件が気に入らないんだけど**
A: 「ここを直して」と言えば直します。「そもそも何を終わらせたいか自体が違ってた」なら段階1に戻ります。何度でも戻れます。むしろ一発で当たると逆に不安になってください。

**Q: 進捗ファイルだけ完成して本体が完成してなかった**
A: それです。「進捗ファイルを完了にする」は **アプリ完成の影武者** であって本体ではありません。影武者しか条件文に書いていないと、評価モデルは影武者だけ満たして満足します。このプラグインの提示する条件文は冒頭に「**そもそも何を終わらせたいか**」を一文で置き、その下に観測可能な証拠を並べる構造にしてあります。もし影武者だけの条件が出てきたら、それはバグです。「目的が抜けてる」と言ってもらえれば直します。

**Q: Constitution 読まないと使えない？**
A: 使えます。スキル側が必要なときに参照します。人間が読むのは趣味の領域です。ただし、なぜこのスキルがこんな動きをするのか気になりだしたら、答えは全部そこに書いてあります。読み始めると止まらないので、寝る前は推奨しません。
