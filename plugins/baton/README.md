# baton

> 「実装できました！」
> 「お、じゃあ実機で確認しとくわ」
> （三日後）
> 「……そういえばあれ、確認したっけ？」

Claude が「終わりました」と言った瞬間、レースはまだ半分です。残りの半周――実機で動かす、本番に反映する、目視で確かめる――は人間というアンカーの区間です。そしてこの中継は、たいてい口頭の一言で済まされます。口頭の一言は、テイクオーバーゾーンで落ちます。手順を一つ飛ばし、見るべきでないところを見て、あるいは確認そのものが忘れられて、バトンは芝生の上を転がったまま、なんとなく本番に乗る。

`baton` は、その落ちるバトンを、書いて手渡すプラグインです。トラック競技で最も速い選手が並んでいても、負けるチームはたいてい、走力ではなく受け渡しで負けます。

## これは何？

Claude の作業が完了しても、Claude 自身には検証できない区間が残ることがあります。デバイスに焼いて動かす。本番のスイッチを入れる。ブラウザで見て「うん、これでいい」と頷く。物理的な結果を確かめる。どれも人間にしか走れない直線です。

問題は、「Claude 完了」と「人間が検証」の間にあるテイクオーバーゾーンです。ここは普通、会話の中の一文で埋められ、そして一文は消えます。`baton` は、この区間に必ずバトンを実体で握らせます。完了時に「あなたがやること」を、コピペできる手順書として書き出す。前提として何が終わっているか、番号付きでそのまま実行できる手順、何を見れば成功と言えるかの確認観点、そして失敗したときの戻し方。この一枚がバトンです。

## 思想: 報告と検証の間には、必ず橋を架ける

「作業完了の報告」と「人間の検証開始」は、別々の出来事です。この二つの間の空白こそ、仕事が静かに失敗する場所です。何かが間違っていたからではなく、バトンパスそのものが行われなかったから。世界記録保持者を四人並べても、握らせ損ねればタイムは記録すらされません。

だから `baton` の立場は単純です。人間の検証が要る作業は、その検証の仕方を人間が思い出しながら組み立て直すのではなく、走り終えた Claude が書いて渡す。完了した作業は、人間が「どう確かめればいいか」を正確に知るまで、まだ完了していません。その知識は、区間を走り終えた Claude の側に握られているのだから、次の走者へ手渡すべきものです。

## 特徴

- **手順は貼れる形で書く**: 「動くか確認して」ではなく、「このコマンドを貼る」「このURLを開く」「この画面のここを見る」。人間が推測で埋める空白を残しません。埋められない手順があるなら、それは作業がまだ渡せる状態にない合図です
- **確認観点は目に見える信号で**: 「たぶん大丈夫」は成功条件になりません。「ログにXが出れば成功」「LEDが緑になれば成功」。ゴールテープが張られていなければ、走者はどこで止まればいいのか分かりません
- **戻し方まで書ききる**: 失敗したとき元へ戻す正確な手順まで書きます。戻せない変更なら、戻せないと正直に書きます。人間には、走り出す前に賭け金を知る権利があります
- **書き出し先は現場に合わせる**: プロジェクトに `notes/` があれば日付つきでそこへ貯め、なければ `.claude/baton.local.md` へ一枚だけ

## なぜフックが無いのか

`baton` にはフックがありません。「作業が完了した」という号砲を、機械が確実に鳴らすのは難しいからです。テストが通ったから完了とは限らないし、Claude が長い返答をしたから完了とも限りません。完了かどうか、そして人間の検証が要るかどうかは、トラックの状況を読んでいる Claude 自身が判断するのが最も正確です。だから完了の検知はスキル、つまり Claude の判断に委ね、機械的なフックは置きません。誤って毎周バトンをせがむプラグインは、すぐに耳を塞がれ、本当に要るときのバトンまで一緒に無視されます。

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/baton:write` | 直前の作業について、前提・手順・確認観点・戻し方を手順書として書き出す |
| Skill | `offer-baton` | 人間の検証が要る作業の完了を見て、バトン作成をその場で促す判断 |

## 使い方

インストールすれば、スキルは黙ってスタートラインで待機します。人間の検証が要る作業を終えたとき、Claude は「終わりました」で止まらず、同じ息で「実機での確認が要ります。手順書を書いておきますか」と、あなたの区間を名指しで手渡そうとします。乗ればコマンドが走り、断ればあなたのやり方で受け取ってください。自分から書かせたいときは `/baton:write` を直接呼べます。特に見てほしい観点があれば、引数で渡せます。

## できないこと・正直な制約

- **バトンは検証を代走しません。** これは「あなたがやること」を書き出すツールであって、実機確認や本番反映そのものを肩代わりするものではありません。橋は架けますが、その先の直線を走るのはあなたです
- **手順の質は、作業の文脈が Claude に見えている範囲を超えません。** 会話に現れなかった前提や、Claude が触れなかった環境の事情は、バトンにも握られません。バトンが的外れなら、それは作業がまだ渡せる状態になっていない兆候かもしれません
- **秘密はバトンに書きません。** バトンはコミットされうるファイルです。トークンや鍵が手順に要るときも、値は書かず、どこで入手するかだけを示します。コミット前に一度目を通してください
- **バトンは走行ログではありません。** 毎回まっさらな一枚です。`.claude/baton.local.md` にフォールバックした場合、前のバトンは上書きされます。周回の記録を残したいなら、`notes/` 側に日付つきで貯まるので、そちらを使ってください

## よくある質問

**Q: Claude が検証まで全部やってくれればよくないですか？**
A: 実機の画面を見る目と、本番のスイッチに責任を取る名前は、まだあなたのものです。バトンはその事実を変えません。渡し方だけを整えます。

**Q: 手順書を読まずに走り出してもいいですか？**
A: 止めません。ただし確認観点の節には「何を見れば成功と言えるか」が書いてあります。読まずに走ったあなたが何を見落としたのかも、あとで読み返せばだいたいそこに書いてあります。

**Q: バトンを落としたらどうなりますか？**
A: 手順書には「失敗したときの戻し方」の節が必ず付いています。落ちた場所から拾い直せるのが、口頭の引き継ぎとの違いです。口頭の一言は、落ちたら芝生に消えます。

速く走ることは、半分でしかない。もう半分は、次の手にきちんと握らせることだ。
