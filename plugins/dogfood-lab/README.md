# dogfood-lab

> 「渾身のスキルを書いた」
> 「description も練った」
> 「よし、これで作業が楽になるぞ」
> 「……ところで、あれ一度でも発火したっけ?」

観察日誌より。被験体「自作スキル」、投入から三週間。作者は毎日それを使っているつもりでいる。しかし記録を取ると、手が伸びているのは別の道具ばかりだった。当のスキルは一度も動いていない。動いていないという事実は、動いていないがゆえにログを残さない。存在しないものは観測できない。本装置は、その観測できないものを観測するために設けられた。

`dogfood-lab` は、あなたのプラグインが本当に生きているかを検死する研究係です。全セッションに羽より軽いロガーを一枚挿し、あなたの手がどのスキル・コマンドを起こしたかを黙って書き留めます。そして頼まれたときだけ台帳を開き、「これは生きている」「これは一度も動いていない」を、証拠つきで突きつけます。

## これは何？

ドッグフーディングしているつもりでも、人はよく馴れた道具にしか手が伸びません。せっかく書いたスキルがトリガー語をほんの少し外していて一度も刺さっていない、という標本は、この研究室で最もありふれた検体です。しかしそのスキルは「呼ばれなかった」という記録を自分で残せないので、あなたは動いていないことに永遠に気づけません。気づけないことにだけは、気づけないのです。

`dogfood-lab` は二枚のフックで、スラッシュコマンドの利用とスキルの発火だけを 1 行の JSONL に書き溜めます。それ以外は何も見ませんし、何も言いません。そして `/dogfood-lab:report` を呼ぶと、溜まった記録を、いま入っているプラグインの全在庫と突き合わせて、生存報告を出します。報告の主役は発火回数の多いスキルではありません。**一度も発火していないもの**です。それこそ、あなたが自力では観測できなかった唯一の標本だからです。

## コンポーネント

| 種別 | 名前 | 役目 |
|------|------|------|
| Hook | `UserPromptSubmit` | プロンプトが `/` で始まるときだけ、コマンド名を 1 行記録する。普通の会話は見ない |
| Hook | `PreToolUse` (Skill) | これから走るスキル名を 1 行記録する |
| Command | `/dogfood-lab:report` | 記録を全在庫と突き合わせ、生存報告を出す |

## 仕組み

三つの部品でできています。研究室らしく、それぞれに役目を割り振ってあります。

### 二枚のロガー — 軽く、そして寡黙に

`UserPromptSubmit` フックは、あなたのプロンプトが `/` で始まるときだけ、そのコマンド名を記録します。普通の会話は一切見ません。プロンプトの中身は、どのプラグインが動いたかを何も語らないからです。`PreToolUse` フックは `Skill` ツールの起動を捉え、これから走るスキル名を記録します。

どちらも規律は同じで、全プロンプト・全スキル呼び出しで走る以上、ほとんど無料でなければなりません。jq を一度呼んで、1 行追記して、抜ける。会話を待たせず、会話に何も出さず、そして絶対にツール呼び出しを失敗させません。観測対象を壊すロガーは、ロガーがいないより悪いのです。だからエラーは全部飲み込んで、黙って exit します。被験体に触れずに観察する、が第一原則です。

記録は `~/.claude/dogfood-lab/events.jsonl` に、人間が読める 1 行 JSON として積まれます。標本を暗室にしまい込むのは方針に反するので、見える場所に置きます。

### 集計 — 発火を在庫と突き合わせる

`/dogfood-lab:report` を呼ぶと、まず記録を集計し、次に「いま入っているプラグイン」の一覧を二つの情報源から組み立てます。設定の `enabledPlugins` と、ディスク上のプラグインキャッシュです。どちらか一方では標本が欠けるので、両方を突き合わせて「発火しえたはずのもの」の全体像に近づけます。

その全在庫と、実際に発火した記録を照合し、プラグインごと・スキルごと・コマンドごとの発火回数と最終発火日を出します。そして在庫にあるのに記録に一度も現れないものを、**一度も発火していない**として台帳の先頭に並べます。生きている検体を数える報告書は世に多いですが、この装置が主役に据えるのは死体のほうです。

### 見立て — 発火すべきだったのに、しなかった機会

報告の最後に、控えめな見立てを添えます。一度も発火していないスキルの description からトリガー語を拾い、直近のトランスクリプトを少しだけサンプリングして、そのトリガー語が実際に会話に現れていたのに発火しなかった痕跡を探します。

これはあくまでベストエフォートの推定です。小さなサンプルへのパターン照合であって、保証ではありません。報告もそう正直に名乗ります。それでも「トリガー語がそこにあったのにスキルが動かなかった」は、プラグイン作者にとって最も価値が高く最も見えにくい失敗なので、当てにいく価値があります。検死報告に「おそらく」と書くのは、推測を隠すためではなく、推測だと明かすためです。

## 使い方

インストールすれば、二枚のロガーは黙って働き始めます。あなたは普段どおり作業を続けてください。しばらく使ったころに `/dogfood-lab:report` を呼ぶと、そこまでの生存報告が出ます。

引数で観点を絞ることもできます。`/dogfood-lab:report このプロジェクトだけ` のように、どこを強調してほしいかを添えられます。

記録が空のうちは、報告も正直に「まだ何も溜まっていない」と言います。ロガーは、このプラグインを入れて使い始めてからの分しか記録できません。観察は投入の瞬間から始まるので、過去はさかのぼれません。

## 記録の置き場所とローテーション

記録は `~/.claude/dogfood-lab/events.jsonl` に積まれます。`report` を実行したとき、このファイルが 10MB を超えていたら、タイムスタンプ付きの名前で脇に退避させ、新しい空のログに切り替えます。古い台帳は消えず、読める形でディスクに残ります。標本は捨てない主義です。

## よくある質問

**Q: 過去のセッションもさかのぼって集計してくれる？**
A: できません。ロガーはインストールして使い始めてからの発火しか見ていません。投入前にどれだけ発火していたかは、この装置には永遠に分かりません。観察していないものは、観察していないのです。

**Q: `/dogfood-lab:report` に出てこないプラグインは、遊んでいるということ？**
A: 断定はできません。ロガーが捉えるのはスラッシュコマンドの利用とスキルの発火だけです。価値がフックやエージェントにあるプラグインは、まともに働いていてもログに一行も残しません。「ログにない」が言えるのは「そのスキルとコマンドが発火しなかった」までで、「そのプラグインが遊んでいた」ではありません。報告もそこは律儀に区別します。

**Q: 「発火すべきだった」の見立ては、どこまで信じていい？**
A: 推定として信じてください。小さなサンプルへのパターン照合なので、見落としも空振りもあります。報告は必ず「推定」と名乗り、当てにしすぎないよう促します。研究係が自分の測定精度を隠したら、それはもう研究ではありません。

**Q: プロンプトに書いた内容は記録されている？**
A: いいえ。記録するのは `/` で始まるコマンドのトークンとスキル名だけです。普通のプロンプト本文はその場で捨てます。観察日誌に書くのは「何が動いたか」であって、「あなたが何を打ったか」ではありません。

## 構成

```
dogfood-lab/
├── .claude-plugin/
│   └── plugin.json
├── hooks/
│   ├── hooks.json          # UserPromptSubmit と PreToolUse(Skill) の二枚
│   └── log.sh              # jq 一回で 1 行 JSONL 追記して即抜ける
├── commands/
│   ├── report.md           # 集計を在庫と突き合わせ生存報告を出す
│   └── report.sh           # 集計・在庫列挙・ローテーションの機械部分
└── README.md
```

## 前提

- `jq` がパスにあること。ロガーも集計も jq で JSON を扱います。
- 発火回数を正しく在庫と突き合わせるには、プラグインが `~/.claude/plugins/cache/` 配下に展開されていること。マーケットプレイス経由でインストールしていれば自然にそうなります。

被験体に触れず、ただ記録する。一度も動かなかった標本にこそ、いちばん長い注釈がつく。
