# dogfood-lab

> 「渾身のスキルを書いた」
> 「description も練った」
> 「よし、これで作業が楽になるぞ」
> 「……ところで、あれ一度でも発火したっけ?」

プラグイン作者には、他の誰にも見えない盲点があります。自作のスキルやコマンドが、実際に発火しているのかどうか。動いていれば手応えがありますが、一度も発火しなかったスキルは、ログにすら残りません。存在しないものは観測できない。動いていない事実にだけは、永遠に気づけないのです。

`dogfood-lab` は、その「発火しなかったもの」を見えるようにする係です。全セッションに羽のように軽いロガーを一枚挿し、あなたの手が実際にどのスキル・コマンドを起こしたかを黙って記録します。そして頼まれたときだけ、記録を集計して「これは生きている」「これは一度も動いていない」を突きつけます。

## これは何？

自分でドッグフーディングしているつもりでも、人はよく使う道具にしか手が伸びません。せっかく作ったスキルが、トリガー語をほんの少し外していて一度も刺さっていない、なんてことは日常茶飯事です。しかしそのスキルは「呼ばれなかった」という記録を自分では残せないので、あなたは動いていないことに気づけません。

`dogfood-lab` は二つのフックで、スラッシュコマンドの利用とスキルの発火だけを 1 行の JSONL として書き溜めます。それ以外は何も見ませんし、何も言いません。そして `/dogfood-lab:report` を呼ぶと、溜まった記録を、いま入っているプラグインの全在庫と突き合わせて、生存報告を出します。

報告の主役は、発火回数の多いスキルではありません。**一度も発火していないもの**です。それこそが、あなたが自力では観測できなかった一枚です。

## 仕組み

三つの部品でできています。

### 二枚のロガー — 軽く、そして寡黙に

`UserPromptSubmit` フックは、あなたのプロンプトが `/` で始まるときだけ、そのコマンド名を記録します。普通の会話は一切見ません。プロンプトの中身は、どのプラグインが動いたかを何も語らないからです。`PreToolUse` フックは `Skill` ツールの起動を捉え、これから走るスキル名を記録します。

どちらも規律は同じです。全プロンプト・全スキル呼び出しで走るので、ほとんど無料でなければなりません。jq を一度呼んで、1 行追記して、抜ける。会話を待たせず、会話に何も出さず、そして絶対にツール呼び出しを失敗させません。観測対象を壊すロガーは、ロガーがいないより悪いからです。エラーは全部飲み込んで、黙って exit します。

記録は `~/.claude/dogfood-lab/events.jsonl` に、人間が読める 1 行 JSON として積まれます。見える場所に置くのが方針です。

### 集計 — 発火を在庫と突き合わせる

`/dogfood-lab:report` を呼ぶと、まず記録を集計し、次に「いま入っているプラグイン」の一覧を二つの情報源から組み立てます。設定の `enabledPlugins` と、ディスク上のプラグインキャッシュです。どちらか一方では不完全なので、両方を突き合わせて「発火しえたはずのもの」の全体像に近づけます。

その全在庫と、実際に発火した記録を照合して、プラグインごと・スキルごと・コマンドごとの発火回数と最終発火日を出します。そして在庫にあるのに記録に一度も現れないものを、**一度も発火していない**として先頭に並べます。

### 見立て — 発火すべきだったのに、しなかった機会

報告の最後に、控えめな見立てを添えます。一度も発火していないスキルの description からトリガー語を拾い、直近のトランスクリプトを少しだけサンプリングして、そのトリガー語が実際に会話に現れていたのに発火しなかった痕跡を探します。

これはあくまでベストエフォートの推定です。小さなサンプルに対するパターン照合であって、保証ではありません。報告もそう正直に名乗ります。それでも、「トリガー語がそこにあったのにスキルが動かなかった」は、プラグイン作者にとって最も価値が高く、最も見えにくい失敗なので、当てにいく価値があります。

## 使い方

インストールすれば、二枚のロガーは黙って働き始めます。あなたは普段どおり作業を続けてください。しばらく使ったころに `/dogfood-lab:report` を呼ぶと、そこまでの生存報告が出ます。

引数で観点を絞ることもできます。`/dogfood-lab:report このプロジェクトだけ` のように、どこを強調してほしいかを添えられます。

記録が空のうちは、報告も正直に「まだ何も溜まっていない」と言います。ロガーは、このプラグインを入れて使い始めてからの分しか記録できません。過去はさかのぼれません。

## 記録の置き場所とローテーション

記録は `~/.claude/dogfood-lab/events.jsonl` に積まれます。`report` を実行したとき、このファイルが 10MB を超えていたら、タイムスタンプ付きの名前で脇に退避させ、新しい空のログに切り替えます。古いデータは消えず、読める形でディスクに残ります。

## 正直な制約

- **ロガーは過去を記録できません。** インストールして使い始めてからの発火しか見えません。それ以前にどれだけ発火していたかは、このプラグインには分かりません。
- **見えるのはスキルとコマンドだけです。** ロガーが捉えるのはスラッシュコマンドの利用とスキルの発火です。価値がフックやエージェントにあるプラグインは、ログに現れません。だから「ログにない」は「そのプラグインが遊んでいる」の証明ではなく、あくまで「そのスキルとコマンドが発火しなかった」までしか言えません。報告もそこは区別します。
- **「発火すべきだった」の見立ては推定です。** 小さなサンプルへのパターン照合なので、見落としも空振りもあります。報告は必ず「推定」と名乗り、当てにしすぎないよう促します。
- **プロンプトの中身は記録しません。** 記録するのは `/` で始まるコマンドのトークンとスキル名だけです。普通のプロンプト本文は捨てます。

## 構成

```
dogfood-lab/
├── .claude-plugin/
│   └── plugin.json
├── hooks/
│   ├── hooks.json          # UserPromptSubmit と PreToolUse(Skill) の二枚
│   └── log.sh              # jq 一回で 1 行 JSONL 追記して即抜ける
├── commands/
│   ├── report.md           # 集計を在庫と突き合わせ生存報告を出す
│   └── report.sh           # 集計・在庫列挙・ローテーションの機械部分
└── README.md
```

## 前提

- `jq` がパスにあること。ロガーも集計も jq で JSON を扱います。
- 発火回数を正しく在庫と突き合わせるには、プラグインが `~/.claude/plugins/cache/` 配下に展開されていること。マーケットプレイス経由でインストールしていれば自然にそうなります。
