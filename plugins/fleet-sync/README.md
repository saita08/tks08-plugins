# fleet-sync

> 「このコーディング規約、A のリポジトリにも入れとこ」
> 「B にもコピペ、C にもコピペ」
> 「……あれ、A のやつ、いつの間にか少し違うぞ」
> 「どっちが正なんだ、これ」

一箇所で丁寧に育てた規約は、コピペで艦隊に配られた瞬間から、少しずつ、しかし確実に散り散りになっていきます。`fleet-sync` は、その散らばりを見張り、母港から配り直す補給部隊です。弾薬と食料の代わりに、規約とフックと設定断片を各艦へ届けます。

## これは何？

プロジェクトをいくつも抱えていると、どの艦にも同じ物資を積みたくなります。共通のコーディング規約、個人 `CLAUDE.md` の共通セクション、どのプロジェクトにも入れたい `settings.json` の断片、汎用のフックスクリプト。制度と呼べる補給物資たちです。

これらは母港で育てるのが正しい。問題は「配る」ところにあります。手でコピペした瞬間、原本と配布先は別々の航路を進み始めます。母港で直しても前線は古いまま。前線でいじれば母港と食い違う。三ヶ月も経てば、どの弾薬が現行仕様なのか誰も分からなくなっています。制度は、育てる場所ではなく補給線が無いせいで死蔵されるのです。

`fleet-sync` は、原本リポジトリを一つの母港と定め、そこから艦隊全体へ制度を届け、そして乖離を見張ります。母港が正、各艦はその補給を受ける側。この関係を宣言しておけば、あとは「今どの艦がズレてる?」と「ズレてる艦に補給し直して」の二言で済みます。

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/fleet-sync:status` | 艦隊全体の在庫を照合し、「配布先 × ファイル」のマトリクスで乖離を報告する。何も書き換えない偵察行動 |
| Command | `/fleet-sync:push` | 古い物資と未着の物資だけを各艦の甲板に積む補給行動。逆輸入候補には指一本触れない |
| Skill | `distribution-layers` | 何が艦隊全体の共通物資で、何がその艦固有の備品かを見分ける判断基準。制度文書の二層モデルを Claude が読む |

## 仕組み

母港リポジトリの `.claude/fleet-sync.local.md` に、何を配るか(`files`)とどこへ配るか(`destinations`)を宣言します。この宣言が艦隊の航路図です。

`/fleet-sync:status` は、宣言された各ファイルを、各配布先の同じ場所と一つずつ照合します。結果は「配布先 × ファイル」のマトリクスで返ってきます。補給済みか、在庫が古いか、まだ届いていないか、甲板を見渡すように一目で分かります。

そして補給部隊が最も神経を使うのが、**前線で誰かが意図して手を加えた形跡**の扱いです。単に古いだけの物資と、その場で意図的に改造されたものは、見た目の diff は同じでも意味がまるで違います。前者はそのまま新品に交換して構いませんが、後者を黙って上書きすれば、現場が下した決定を消してしまいます。`fleet-sync` は後者を「逆輸入候補」として区別し、補給対象から外して、あなたに判断を返します。前線の改造には、母港へ持ち帰る価値があるかもしれないからです。

## 使い方

まず母港リポジトリの `.claude/fleet-sync.local.md` に航路図を書きます。

```markdown
---
files:
  - CONVENTIONS.md
  - .claude/settings-fragment.json
destinations:
  - ~/projects/service-a
  - ~/projects/service-b
  - ~/work/internal-tool
---

# fleet-sync manifest

この母港が艦隊に配る制度の一覧。files は原本ルートからの相対パス、
destinations は各配布先リポジトリのルート。何をなぜ配るのかは、
ここに散文で書き添えておくと、未来の自分が助かる。
```

あとは二つのコマンドだけです。

- `/fleet-sync:status` — 艦隊全体の乖離をマトリクスで偵察する。引数に配布先名を渡せば一隻に絞れます。
- `/fleet-sync:push` — 古い・未着の物資を補給し直す。逆輸入候補はスキップし、判断はあなたに返る。

「何を配るべきで、何は配るべきでないか」——艦隊共通の制度なのか、その艦固有のルールなのか——の判断に迷ったら、同梱の `distribution-layers`(制度文書の二層構造)を Claude が読んで助けてくれます。共通規約は全艦に配るべきですが、一隻のアーキテクチャにしか意味のないルールを艦隊中にばら撒くと、それは補給ではなく汚染です。

## これは補給であって、版管理ではない

はっきりさせておきます。`fleet-sync` はバージョン管理システムではありません。母港の航海日誌を持っているのは、母港リポジトリの git です。この補給部隊が運ぶのは、宣言された共通ファイルの**今の状態**を母港から前線へ、それだけです。過去の航路を突き合わせたり、失われた寄港地を再現したりはしません。

だからこそ、逆輸入候補の判定は「証拠を重ねた推測」であって、確実な判定ではありません。配布先が git 管理下なら履歴を手がかりにできますが、そうでなければタイムスタンプや diff の中身から見当をつけるしかない。判断がつかないときは、`fleet-sync` は必ず「逆輸入候補」の側に倒します。古い物資を一周期長く放置する損失は取り返せますが、現場の改造を上書きした損失は取り返せないからです。

## よくある質問

**Q: `push` したら、配布先が勝手にコミットされたりしませんか?**
A: しません。補給部隊の仕事は各艦の甲板に物資を積むところまでです。それを航海日誌に記すか、いつ記すかは、その艦の艦長が決めることです。`fleet-sync` は `git add` も commit も push もしません。配布先はそれぞれ独自の履歴とリリースの都合を持つ、誰かのリポジトリだからです。

**Q: 配布先を git で管理していません。動きますか?**
A: 動きます。git ならタイムスタンプの怪しい写しを逆輸入候補として見分ける精度が上がりますが、ただのディレクトリでも補給はできます。git は補給の前提ではなく、偵察の精度を上げる装備です。

**Q: 母港にある他のファイルも、ついでに配ってくれませんか?**
A: 配りません。航路図に宣言したものだけが補給対象です。マニフェストが艦隊の共有範囲の境界であり、宣言されていない物資を勝手に前線へ送る補給部隊は、補給部隊ではなく密輸業者です。

**Q: 「逆輸入候補」って、結局どっちに転ぶんですか?**
A: 迷ったら必ず「触らない」側に転びます。古い物資は次の補給でいつでも入れ替えられますが、上書きで消えた現場の判断は二度と戻りません。この非対称が、判定に迷ったときの答えを決めています。取り返せる損失より、取り返せない損失を恐れる。それが補給部隊の臆病さであり、規律です。

## 前提

- `diff` と基本的なシェルが使えること。特別な依存はありません。
- 母港リポジトリのルートに `.claude/fleet-sync.local.md` があること。無ければ `/fleet-sync:status` は航路図が無いと言って停泊したまま動きません。

物資は母港で育て、艦隊で配れ。散らばりは、配る仕組みが無い艦隊にだけ訪れる。
