# fleet-sync

> 「このコーディング規約、A のリポジトリにも入れとこ」
> 「B にもコピペ、C にもコピペ」
> 「……あれ、A のやつ、いつの間にか少し違うぞ」
> 「どっちが正なんだ、これ」

一箇所で丁寧に育てた規約は、コピペで艦隊に配られた瞬間から、少しずつ、しかし確実に散り散りになっていきます。`fleet-sync` は、その散らばりを見張り、原本から配り直す係です。

## これは何？

プロジェクトをいくつも持っていると、どのリポジトリにも同じであってほしいものが出てきます。共通のコーディング規約、個人 `CLAUDE.md` の共通セクション、どのプロジェクトにも入れたい `settings.json` の断片、汎用のフックスクリプト。制度と呼べるものたちです。

これらは一箇所で育てるのが正しい。問題は「配る」ところにあります。手でコピペした瞬間、原本と配布先は別々の道を歩き始めます。原本を直しても配布先は古いまま。配布先をいじれば原本と食い違う。三ヶ月後には、どれが正なのか誰も分からなくなっています。制度は、育てる場所ではなく配る仕組みが無いせいで死蔵されるのです。

`fleet-sync` は、原本リポジトリを一つの母港と定め、そこから艦隊全体へ制度を配り、そして乖離を見張ります。原本が正、配布先はその写し。この関係を宣言しておけば、あとは「今どれがズレてる?」と「ズレてるやつを配り直して」の二言で済みます。

## 仕組み

原本リポジトリの `.claude/fleet-sync.local.md` に、何を配るか(`files`)とどこへ配るか(`destinations`)を宣言します。この宣言が艦隊の地図です。

`/fleet-sync:status` は、宣言された各ファイルを、各配布先の同じ場所と一つずつ照合します。結果は「配布先 × ファイル」のマトリクスで返ってきます。同期済みか、古くなっているか、まだ届いていないか、一目で分かります。

そして `fleet-sync` が最も気を遣うのが、**配布先で誰かが意図して手を加えた形跡**の扱いです。単に古いだけの写しと、その場で意図的に書き換えられたものは、見た目の diff は同じでも意味がまるで違います。前者は上書きして構いませんが、後者を黙って上書きすれば、誰かの決定を消してしまいます。`fleet-sync` は後者を「逆輸入候補」として区別し、上書き対象から外して、あなたに判断を返します。

`/fleet-sync:push` は、古い写しと未着のファイルだけを配布先に書き込みます。逆輸入候補には指一本触れません。そして配布は**配布先の作業ツリーに書くところまで**。コミットはしません。配布先はそれぞれ独自の履歴とリリースの都合を持つ、誰かのリポジトリだからです。いつ履歴に取り込むかは、その持ち主が決めることです。

## 使い方

まず原本リポジトリの `.claude/fleet-sync.local.md` に地図を書きます。

```markdown
---
files:
  - CONVENTIONS.md
  - .claude/settings-fragment.json
destinations:
  - ~/projects/service-a
  - ~/projects/service-b
  - ~/work/internal-tool
---

# fleet-sync manifest

この母港が艦隊に配る制度の一覧。files は原本ルートからの相対パス、
destinations は各配布先リポジトリのルート。何をなぜ配るのかは、
ここに散文で書き添えておくと、未来の自分が助かる。
```

あとは二つのコマンドだけです。

- `/fleet-sync:status` — 艦隊全体の乖離をマトリクスで見る。引数に配布先名を渡せば一隻に絞れます。
- `/fleet-sync:push` — 古い・未着のファイルを配り直す。逆輸入候補はスキップし、判断はあなたに返る。

「何を配るべきで、何は配るべきでないか」——共通の制度なのか、そのプロジェクト固有のルールなのか——の判断に迷ったら、同梱の判断基準(制度文書の二層構造)を Claude が読んで助けてくれます。

## これは同期であって、版管理ではない

はっきりさせておきます。`fleet-sync` はバージョン管理システムではありません。原本の履歴を持っているのは、原本リポジトリの git です。このプラグインが運ぶのは、宣言された共通ファイルの**今の状態**を原本から外へ、それだけです。履歴を突き合わせたり、コミットを再生したりはしません。

だからこそ、逆輸入候補の判定は「証拠を重ねた推測」であって、確実な判定ではありません。配布先が git 管理下なら履歴を手がかりにできますが、そうでなければタイムスタンプや diff の中身から見当をつけるしかない。判断がつかないときは、`fleet-sync` は必ず「逆輸入候補」の側に倒します。古い写しを一周期長く放置する損失は取り返せますが、意図した編集を上書きした損失は取り返せないからです。

## できないこと

- **コミット・プッシュはしません。** 配布は配布先の作業ツリーに書くところまで。履歴に入れるのは持ち主の仕事です。
- **配布先が git であることを前提にしません。** git なら逆輸入候補の判定精度が上がりますが、ただのディレクトリでも動きます。
- **宣言していないものは配りません。** マニフェストが艦隊の共有範囲の境界です。原本にある他のファイルには触れません。
- **配布先のリポジトリを勝手に増やしません。** 艦隊の地図はあなたの宣言です。

## 前提

- `diff` と基本的なシェルが使えること。特別な依存はありません。
- 原本リポジトリのルートに `.claude/fleet-sync.local.md` があること。無ければ `/fleet-sync:status` は地図が無いと言って止まります。
