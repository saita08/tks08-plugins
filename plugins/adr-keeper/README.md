# adr-keeper

> 「このプロジェクト、1機能1PRでいこう」
> 「了解です」
> ――数日後、別のセッションにて――
> 「この3つの変更、まとめて1つのPRにしておきますね」
> 「!!!」

先週きっちり決めたはずの方針を、今週のClaudeが悪びれもせず逆走する。悪気はありません。決めた瞬間の会話は、そのセッションと一緒に蒸発していて、今のClaudeはそれを一度も見ていないのです。`adr-keeper` は、決定が下されたその席に控えていて、言質を議事録に清書し、後日「言った言わない」が起きない棚に綴じておく番人です。口約束を、日付と理由のついた公文書に変える係、と言ってもいい。

## これは何？

会話の中では、いろいろな方向性が決まります。「この層がこの責務を持つ」「このフォーマットで統一する」「速さより設定の簡潔さを取る」。どれもその場ではちゃんと理由があって議決されている。ところがその議事は口頭のまま、チャットログの中にしか残らず、次のセッションのClaudeはその議場にいなかったので、平気で逆の動議を出してきます。決定が「記録として綴じられていない」せいで、同じ審議が何度も蒸し返され、方針がふらつくのです。

`adr-keeper` は、この蒸発を止めます。方向性の決定が下されたその場で ADR(Architecture Decision Record)として起票し、以後のすべてのセッションが閲覧できる書庫――リポジトリのルート直下の `adr/`――に綴じます。ADR は「何を決めたか」だけでなく「なぜそう決めたか」を書き残すので、未来のセッションは議決をなぞるだけでなく、その理由まで受け継ぎます。理由が分かれば、盲目的に逆走することはなくなります。前任者が何を考えて署名したかが読めれば、後任は軽々しく破棄しません。

## ADR とは

Architecture Decision Record は、一つの決定を短い標準フォーマットで書き残す、世界的に定着した習慣です。四つの節でできています。

- **Status** ― その決定が今も有効か(Accepted / Proposed / Superseded など)。効力の切れた条項が生きているふりをしないための欄です。
- **Context** ― なぜ決める必要があったのか。当時どんな制約と力が働いていたのか。ここが一番大事な節です。理由はコードには残らないので、未来の読み手が「その前提は今も成り立つか」を判断できるのは、この節があるからです。
- **Decision** ― 何を決めたか。一つの決定を、はっきりと。
- **Consequences** ― その結果、何が楽になり、何が難しくなったか。正直に両面を。受け入れたコストも隠さずに。都合の悪い条項を小さな文字で隠す番人は、番人ではありません。

`adr-keeper` はこの標準形式で起票します。ただし、すでに ADR を運用しているプロジェクトでは、そのプロジェクトの形式・連番・置き場所に従います。既存の兄弟たちと揃っていることのほうが、このプラグインの流儀に従うことより大事だからです。書式の揃っていない記録は、書庫の中で行方をくらますのと同じで、探す者はそこにあると気づけません。

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/adr-keeper:record` | この会話で下された方向性の決定を ADR として起票する |
| Command | `/adr-keeper:list` | 既存の ADR を番号・タイトル・状態つきで一覧する |
| Skill | `adr-guardian` | 議決が下された瞬間を聞き分け、ADR にすべきかを判断する知識ベース |

## 使い方

**`/adr-keeper:record <決定の要約>`** で決定を起票します。要約を渡してもいいし、省略すれば直前の会話から決定を拾い、「これで合ってますか」と確認してから清書します。番号は既存の最大+1、置き場所とフォーマットは既存の運用に合わせます。番人は勝手に連番を振り直したりしません。

**`/adr-keeper:list`** で既存の ADR を一覧します。番号・タイトル・状態が並ぶので、「また決める」前に「もう決まっていないか」を確かめられます。ある決定が別の決定を上書きしている(superseded)関係も見えるようにします。効力を失った古い条項が、生きた条項の顔をして紛れ込まないように。

`adr-guardian` スキルも同梱しています。あなたがコマンドを呼ばなくても、会話の中で方向性の決定が下された瞬間――選択肢の議論が収束し、あなたが「これでいく」と宣言したとき――Claude が「今の、ADR にしておきますか」と控えめに一言添えることがあります。乗らなければ無視して構いません。番人は議事を勝手に清書せず、必ず一度あなたの署名を仰ぎます。

## すべての決定を記録するわけではない

これは大事な設計思想です。ADR は「注意」という通貨を消費します。書庫に綴じられる記録が増えるほど、未来の読み手が目を通す量も増える。だから `adr-keeper` が拾うのは「未来のセッションが、知らずに逆走したら害になる決定」だけです。

拾うのは、後の作業を縛る方向性の決定――構造の選択、プロジェクトの規約、明示的に受け入れたトレードオフ、そして過去の決定の覆し。拾わないのは、タスクから自然に決まる実装上の選択、可逆な実験、既存のルールがすでに定めていること、コードを読めば分かる事実、チェックで機械的に強制できる制約。後者を ADR にすると、本当に大事な決定が些末な記録の山に埋もれて、誰も書庫を開かなくなります。あらゆる立ち話を公文書にする番人のもとでは、公文書がただの紙束になる。記録しないことが正しい場合がある、というのがこのプラグインの節度です。

## よくある質問

**Q: 起票したらコミットまでしてくれる?**
A: しません。番人の仕事は議事を書庫に綴じるところまでで、それを外部に公示するのはあなたの領分です。いつコミットするかはあなたが決めます。

**Q: 会話に理由が書いてなかったら、もっともらしい理由を補ってくれる?**
A: 補いません。番人が証言を捏造したら、それはもう記録ではなく偽造です。Context に書くのは、会話に実在した理由だけ。分からないことは「分からない」と正直に空けます。

**Q: 決めるのを迷っています。代わりに決めてくれる?**
A: それは番人の管轄外です。ADR は下された決定を書き残す道具であって、何を決めるべきかを決める道具ではありません。動議を出すのはあなた、それを議事録にするのが番人です。

**Q: 一つの会話で二つ決まったら、まとめて1本にしてくれる?**
A: 分けます。1記録につき1決定が原則です。混ぜて綴じると、後で探すのも、片方だけを上書きするのも難しくなる。二つの決定は、二つの書類です。

番人は、決まったことしか綴じません。そして、決まったことは必ず綴じます。
