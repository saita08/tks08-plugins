# librarian

> 「その仕様、CLAUDE.md に書いといて」
> 「README にも書いておきますね」
> 「docs/ にも同じ説明があった気がする」
> 「三箇所とも、微妙に違うことが書いてありますが」
> 「……どれが本当？」

三冊の本に三通りの真実が書いてある図書館を、人は図書館と呼びません。呼ぶとしたら、物置です。`librarian` は、その物置に司書を常駐させるプラグインです。

## これは何？

プロジェクトの書かれた知識を、それぞれ性質の違う四つの棚と一枚の案内板に配架する司書です。

- **CLAUDE.md** は館の憲章です。価値観と判断基準だけを書きます。コードベースを明日ゼロから書き直しても生き残る文だけが、ここに住む資格を持ちます
- **docs/** は現在の蔵書です。「今この瞬間に何があるか」だけを書きます。構成、手順、データ契約。読むのはコードを書く前で、書いた後の言い訳のためではありません
- **adr/** は決定の書庫です。「なぜそう決めたか、何を捨てたか」を時系列で綴じます。一度採用された記録は書き換えず、判断が変わったら新しい一枚で上書きではなく置き換えます。書庫はルート直下に建ちます。決定の記録は現状説明の押し入れに仕舞う物ではないからです
- **references/** は標本室です。スプレッドシートのサンプル行、捕獲したAPIレスポンス。散文にせず、加工せず、コードからも読み込まない生データの棚です
- **README.md** は玄関の案内板です。初めて来た人に、ここが何の館で、どの棚に何があるかを示します

配架の原則はただ一つ、**一つの事実に、住所は一つ**。同じ内容が二つの棚にあると、現実が動いたときにどちらか一方だけが更新され、読者はもうどちらも信じられなくなります。腐った文書は、信頼されているぶん、無い文書より悪いのです。

## 特徴

- **配架には必ず証拠を添えます**: 「このファイルのこの節は、書き直し後も生き残る価値観ではなく今日の手順なので、docs/ 行きです」。ファイル名の雰囲気で本を動かす司書は、司書ではなく引っ越し業者です
- **診断と整理は別の動作です**: audit はまず配架案を報告し、あなたが頷いた項目だけを動かします。移動は `git mv` で履歴ごと運び、リンクと索引も同じ一手で追随させます。リンク切れを残す移動は、移動ではなく遺失です
- **綺麗なプロジェクトには綺麗と言います**: 違反ゼロの監査報告は空振りではなく成果です。仕事を見せるために問題を発明したりはしません
- **空の棚は作りません**: 生データの無いプロジェクトに references/ は建てません。空の標本室は家具ではなく、ただの埃です

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/librarian:audit` | 全文書を四層に写像し、違反を証拠つきで報告。承認された配架案だけを実行する |
| Command | `/librarian:setup` | 文書構造の無いプロジェクトに、四層の棚と案内板を実際の観察に基づいて敷設する |
| Skill | `four-track-contract` | 四層知識契約の教義・配架手順・雛形。「この情報どこに書く?」に随時答える |

## 使い方

散らかった館には監査を。

```
/librarian:audit
```

更地には敷設を。

```
/librarian:setup
```

会話の途中で「これって docs/ と README どっちに書くべき?」と聞くだけでも、スキルが教義から答えます。コマンドの儀式は要りません。

## setup が最初にやる、少し行儀のいいこと

敷設のとき、司書は最初の二冊の決定記録を自分で書きます。「ADR を使うという決定」と「四層契約を採用するという決定」です。日付は敷設した当日、却下した代替案も正直に記します。「決定を記録せよ」と説く構造が自分の誕生の記録を持っていなかったら、その館は開館初日から自己矛盾で軋むからです。

## よくある質問

**Q: 全部 CLAUDE.md に書いてはだめですか？**
A: だめではありません。ただし価値観と現状と判断理由を一冊に綴じると、現状が変わるたびに理由が消えるか、理由を守って現状の記述が腐るか、必ずどちらかが起きます。三年もののCLAUDE.mdの中で、今も真実の段落がどれか当てるゲームをしたことがある人は、この分割の理由をすでに体で知っています。

**Q: adr/ がルートにあるのはなぜですか。docs/adr/ のほうが片付いて見えます。**
A: 片付いて見えるのは、決定の記録を現状説明の子分として扱っているからです。「今どうなっているか」と「なぜそうしたか」は親子ではなく、対等な別トラックです。書庫を書斎の押し入れに入れる図書館はありません。

**Q: 勝手にファイルを動かしませんか？**
A: 動かしません。司書は配架案をカウンターに置き、あなたが頷いた分だけ棚に触れます。二つの棚に等しく収まりうる本は、勝手に分類せず、あなたに尋ねます。本の意味を決めるのは司書の仕事ではないからです。

**Q: 一度整理したら、もう散らからないですよね？**
A: 散らかります。図書館は、開館している限り散らかり続けます。だから司書は常勤で雇うのです。大きな変更のあとに、また audit を呼んでください。

**Q: 私のプロジェクトは小さくて、四層も要らない気がします。**
A: 正しい感覚です。この契約は「知識があるならここに住む」を定めるもので、棚を埋めるために知識を製造しろとは言いません。二枚のファイルで足りる館には、二枚のファイルと、それがどこにあるかの案内板だけで十分です。

## できないこと・正直な制約

- **本の中身の正しさまでは保証しません。** 司書が見るのは配架です。docs/ の手順が古いままかどうかは、audit が索引と実体のずれとして気づける範囲でしか検出できません
- **分類は最終的に人間の判断です。** 価値観と現状の境界には、本当に曖昧な本が存在します。司書は基準と推薦を示しますが、票は持ちません
- **公開URLや外部からの参照までは追跡できません。** リポジトリの外から特定パスへリンクが張られている場合、移動の影響は配架案の中で警告するにとどまります

---

良い図書館とは、本がすぐ見つかる場所のことではありません。返す場所が、一つに決まっている場所のことです。
