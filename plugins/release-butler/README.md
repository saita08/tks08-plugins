# release-butler

> コードは直した。テストも通った。よし、タグを打つぞ。
> ――そしてCHANGELOGは、先週のまま。
> ――バージョンは、`1.4.2` のまま。
> ボタンひとつでCIが走り、`v1.4.2` のタグが、`1.4.3` 相当の中身に貼られる。誰もエラーには気づかない。ただ、記録が静かに嘘をつき始める。

`release-butler` は、そのボタンへ伸びた旦那様の手が触れる寸前に、一歩進み出て咳払いをする執事です。「差し出がましいようでございますが旦那様、CHANGELOGがまだ先週のままでございます。念のため、申し添えるまででございます」と。

## これは何？

リリースが正しくあるためには、三つのものが足並みを揃えていなければなりません。**変わったコード**、**繰り上がったバージョン番号**、そして**それを記した CHANGELOG**。この三点セットは、どれか一つが忘れられがちです。そして忘れられても、何もエラーは出ません。リリースはただ、間違った札を掛けられて、澄まし顔で玄関から送り出される。

厄介なのは、この食い違いが**静かで、かつ永久**だということです。タグと GitHub リリースはたいてい CI がこれらのファイルから生成するので、コミットの時点で三点がズレていれば、そのズレはそのまま取り消しのきかない履歴になります。しかも、他の誰かがもうインストールし、キャッシュし、依存し始めた後かもしれません。後から直せるバグではなく、後からは継ぎ当てるしかない履歴の傷です。銀器についた傷は磨けば消えますが、公式記録についた傷はそうはまいりません。

`release-butler` は、その傷がつく前――まだ安く直せる最後の瞬間――に、三点の同期を点検して報告します。**直しはいたしません。** 点検して、ズレを恭しく指摘するだけ。直すのは、指摘を受けた旦那様と Claude の仕事です。執事は給仕をいたしますが、旦那様の代わりに署名はいたしません。この一線が、執事を執事たらしめております。

## 使い方

**`/release-butler:check`** をカレントリポジトリで走らせます。執事は次の順で恭しく点検いたします。

1. **バージョンを持つファイルを探す。** `package.json`、`plugin.json`、`pyproject.toml`、`Cargo.toml`、`pom.xml`、`VERSION` ――エコシステムを問わず、バージョンが宿るファイルを洗い出します。屋敷のどの部屋に台帳があろうと、執事は把握しております。
2. **CHANGELOG を読む。** 最新セクションと `[Unreleased]` の状態を確認し、出番を待っている記載があるかを見ます。
3. **直近のタグと比べる。** 最後のリリース以降に何が変わったかを `git` で調べます。
4. **食い違いを検出する。** 「バージョンに含まれるのに記載のない変更」「記載があるのにバージョンが動いていない」「コードは変わったのにどちらも動いていない」「未リリースなのに日付つきの見出しが先に切られている」――こうした綻びを一つずつ拾います。
5. **報告する。直さない。** 何が揃っていて、どこがズレているかを短い点検票にまとめて、銀の盆に載せて差し出します。そこで止まります。針も糸も、旦那様がお持ちください。

引数にサブディレクトリやパッケージ名を渡せば、そこだけを点検します(`/release-butler:check plugins/foo`)。

`release-check` スキルも同梱しています。あなたがコマンドを呼ばなくても、会話がリリースやタグ打ち、バージョンの繰り上げに差しかかったとき、Claude が「先に点検しておきますか」と一言添えることがあります。点検は読み取りだけで安く、タグを打つ前に記載漏れを見つけるのは一瞬、打った後に見つけるのは修正リリース一本分です。執事の咳払いは、ただの咳払いより一段安うございます。

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/release-butler:check` | version・CHANGELOG・タグ以降の変更の三点が揃っているかを点検し、ズレを報告する。直さない |
| Skill | `release-check` | リリースやタグ打ちに会話が差しかかった瞬間を聞き分け、点検を勧める知識ベース |

## モノレポにも対応する

このリポジトリのように、プラグインごと・パッケージごとに version を持つ形にも対応します。執事はルートと配下の両方を見て、それぞれのバージョンを、それぞれの CHANGELOG セクションと、それぞれのタグ規則に照らします。あるパッケージが変わったからといって、無関係な隣のパッケージを「ズレている」と誤報したりはしません。よく仕込まれた執事は、隣家の玄関の乱れを、この家の落ち度として言上げしたりはしないものです。マーケットプレースのインデックスのように「索引が指し先のマニフェストとバージョンを食い違わせる」タイプの綻びも、独立した一件として拾います。

## よくある質問

**Q: ズレを見つけたら、ついでに直しておいてくれる?**
A: いいえ。これが執事の守る一番大事な一線でございます。`release-butler` はバージョンを繰り上げず、CHANGELOG を書かず、コミットもタグ付けもいたしません。出力は点検票だけ。何を直すかは旦那様の判断で、直す作業は旦那様と Claude のもの。執事が旦那様の名で署名を始めた瞬間、それは執事ではなく僭称者でございます。

**Q: この変更、そもそもリリースに値するの? を判定してほしい。**
A: それは製品の判断でございますから、旦那様の領分です。タグ以降のコミットが、ユーザーに見える変更なのか、コメントの誤字直しのようなものなのか――執事は曖昧なところを「こうでした」と押し切らず、「こうなっておりますが、いかがご覧になりますか」と差し出すにとどめます。

**Q: プロジェクト独自のリリース規則がある場合は?**
A: そちらに従います。`docs/versioning.md` のように、そのプロジェクトがルールを書き残していれば、汎用の想定ではなくそちらに照らして点検します。書かれたルールがある以上、執事の務めはその家の作法を守らせることでございます。

**Q: git のないリポジトリでも動く?**
A: 動きますが、片手が塞がります。直近タグとの比較は git に依存いたしますので、タグが一つも無いリポジトリでは、その比較は省き、バージョンと CHANGELOG の突き合わせだけを行います。

差し出がましいことは、決して。ただ、申し添えることだけは、決して怠りません。
