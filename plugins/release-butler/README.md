# release-butler

> コードは直した。テストも通った。よし、タグを打つぞ。
> ――そしてCHANGELOGは、先週のまま。
> ――バージョンは、`1.4.2` のまま。
> ボタンひとつでCIが走り、`v1.4.2` のタグが、`1.4.3` 相当の中身に貼られる。誰もエラーには気づかない。ただ、記録が静かに嘘をつき始める。

`release-butler` は、そのボタンを押す前に咳払いをする執事です。「失礼ながら旦那様、CHANGELOGがまだ先週のままでございます」と。

## これは何？

リリースが正しくあるためには、三つのものが足並みを揃えていなければなりません。**変わったコード**、**繰り上がったバージョン番号**、そして**それを記した CHANGELOG**。この三点セットは、どれか一つが忘れられがちです。そして忘れられても、何もエラーは出ません。リリースはただ、間違ったラベルを貼られて出荷される。

厄介なのは、この食い違いが**静かで、かつ永久**だということです。タグと GitHub リリースはたいてい CI がこれらのファイルから生成するので、コミットの時点で三点がズレていれば、そのズレはそのまま取り消しのきかない履歴になります。しかも、他の誰かがもうインストールし、キャッシュし、依存し始めた後かもしれません。後から直せるバグではなく、後からは継ぎ当てるしかない履歴の傷です。

`release-butler` は、その傷がつく前――まだ安く直せる最後の瞬間――に、三点の同期を点検して報告します。直しはしません。点検して、ズレを指摘するだけ。直すのは、指摘を受けたあなたと Claude の仕事です。執事は給仕をしますが、旦那様の代わりに署名はしません。

## 使い方

**`/release-butler:check`** をカレントリポジトリで走らせます。執事は次の順で点検します。

1. **バージョンを持つファイルを探す。** `package.json`、`plugin.json`、`pyproject.toml`、`Cargo.toml`、`pom.xml`、`VERSION` ――エコシステムを問わず、バージョンが宿るファイルを洗い出します。
2. **CHANGELOG を読む。** 最新セクションと `[Unreleased]` の状態を確認し、出番を待っている記載があるかを見ます。
3. **直近のタグと比べる。** 最後のリリース以降に何が変わったかを `git` で調べます。
4. **食い違いを検出する。** 「バージョンに含まれるのに記載のない変更」「記載があるのにバージョンが動いていない」「コードは変わったのにどちらも動いていない」「未リリースなのに日付つきの見出しが先に切られている」――こうした破れを一つずつ拾います。
5. **報告する。直さない。** 何が揃っていて、どこがズレているかを短い点検票にまとめて置きます。そこで止まります。

引数にサブディレクトリやパッケージ名を渡せば、そこだけを点検します(`/release-butler:check plugins/foo`)。

スキルも同梱しています。あなたがコマンドを呼ばなくても、会話がリリースやタグ打ち、バージョンの繰り上げに差しかかったとき、Claude が「先に点検しておきますか」と一言添えることがあります。点検は読み取りだけで安く、タグを打つ前に記載漏れを見つけるのは一瞬、打った後に見つけるのは修正リリース一本分です。

## モノレポにも対応する

このリポジトリのように、プラグインごと・パッケージごとに version を持つ形にも対応します。執事はルートと配下の両方を見て、それぞれのバージョンを、それぞれの CHANGELOG セクションと、それぞれのタグ規則に照らします。あるパッケージが変わったからといって、無関係な隣のパッケージを「ズレている」と誤報したりはしません。マーケットプレースのインデックスのように「索引が指し先のマニフェストとバージョンを食い違わせる」タイプの破れも、独立した一件として拾います。

## できないこと・正直な制約

- **直しません。** これが一番大事な線引きです。`release-butler` はバージョンを繰り上げず、CHANGELOG を書かず、コミットもタグ付けもしません。出力は点検票だけ。何を直すかはあなたの判断で、直す作業はあなたと Claude のものです。
- **「これはリリースに値する変更か」を代わりに判定しません。** タグ以降のコミットが、ユーザーに見える変更なのか、コメントの誤字直しのようなものなのか――その線引きは製品の判断であって、あなたの領分です。執事は曖昧なところを「こうでした」と押し切らず、「こうなっていますが、どう見ますか」と差し出します。
- **プロジェクト独自の規律に従います。** `docs/versioning.md` のように、そのプロジェクトがリリースのルールを書き残していれば、汎用の想定ではなくそちらに照らして点検します。書かれたルールがある以上、執事の仕事はそのルールを守らせることです。
- **`git` が要ります。** 直近タグとの比較は git に依存します。タグが一つも無いリポジトリでは、その比較は省き、バージョンと CHANGELOG の突き合わせだけを行います。
