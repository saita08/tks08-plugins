# Release Check

A release quietly breaks when one of three things is forgotten: the code changed, the version was bumped, and the CHANGELOG was updated. When these three fall out of step, the tag stops matching what the tag actually contains — and because tags and releases are usually generated by CI from these files, the discrepancy becomes permanent history the moment it is committed. This command is the butler that inspects the three before the release, and reports what is out of sync. It does not fix anything; the fix belongs to the user and to Claude acting on the report.

Read `${CLAUDE_PLUGIN_ROOT}/skills/release-check/references/sync-model.md` before reporting, for what "in sync" means and the common ways it breaks. Read `${CLAUDE_PLUGIN_ROOT}/skills/release-check/references/version-files.md` for how to detect and read version across the different ecosystems.

If `$ARGUMENTS` names a subdirectory or package, scope the check to it. Otherwise check the whole repository, including nested packages if it is a monorepo.

## Procedure

### 1. Find the version-bearing files

Locate every file that carries a version, using `references/version-files.md` as the catalog. Common ones: `package.json`, `plugin.json` / `.claude-plugin/plugin.json`, `pyproject.toml`, `Cargo.toml`, `composer.json`, `*.gemspec`, `build.gradle`, `pom.xml`, `VERSION`. Use `Glob` to find them. In a monorepo, expect several — check the root and each package under it, and keep track of which version belongs to which.

### 2. Find and read the CHANGELOG

Locate `CHANGELOG.md` (or `CHANGES`, `HISTORY.md`, `NEWS`). Read its top: the most recent version section and any `## [Unreleased]` section. Note the latest documented version and whether there are entries sitting under Unreleased waiting to be released. In a monorepo, a package may have its own changelog or the root may hold a shared one — check both patterns.

### 3. Compare against the last tag

Determine the most recent release tag and what changed since it:

```
git describe --tags --abbrev=0 2>/dev/null
git log --oneline "$(git describe --tags --abbrev=0 2>/dev/null)"..HEAD 2>/dev/null
git diff --stat "$(git describe --tags --abbrev=0 2>/dev/null)"..HEAD 2>/dev/null
```

If the repository has no tags yet, say so — the comparison against a prior release cannot run, but the version-vs-changelog checks still can. In a monorepo with per-package tags, match the tag pattern to the package (e.g. `pkg-name-v1.2.3`) where the project uses one.

### 4. Detect the mismatches

Report each of these that holds, per package where relevant:

- **Code changed since the tag, but the version did not move.** Substantive commits since the last tag while the version file still reads the tagged version — a release would ship changes under the old number.
- **Code changed since the tag, but the CHANGELOG has no matching entry.** Changes with nothing under Unreleased or the latest section describing them — users would get changes the changelog never mentions.
- **The CHANGELOG describes changes, but the version did not move.** Entries written without a corresponding bump.
- **The version moved, but the CHANGELOG did not.** A bumped version whose changes are undocumented.
- **A version-and-date heading was committed for an unreleased change.** A `## [X.Y.Z] - YYYY-MM-DD` section that pre-commits a decision the release step owns, where an `## [Unreleased]` section was expected. Flag it, since the date will almost certainly be wrong by ship time and CI may read it as the release.
- **Version files disagree with each other.** In a monorepo, two files that should share a version but do not — or, in the marketplace-index case, an index whose declared version drifts from the manifest it points to.

### 5. Report, do not fix

Present the findings as a short readiness report: what is in sync, and each mismatch with the specific files and the specific gap. Where a mismatch is ambiguous — a commit since the tag that may or may not be user-visible — say what you found and let the user judge, rather than asserting a verdict. Then stop. Do not edit version files, do not write CHANGELOG entries, do not bump anything, do not commit or tag. The report is the deliverable; acting on it is a separate, user-directed step.

## Constraints

- Report only. This command never edits, bumps, writes changelog entries, commits, or tags. Its entire output is the readiness report.
- The `git` and file reads are inspection only. Nothing here changes the working tree.
- When something is genuinely ambiguous (is this commit user-visible?), surface the ambiguity rather than resolving it silently. The user owns the judgment of what counts as a releasable change.
- Adapt to the project's actual conventions. If it keeps a `docs/versioning.md` or documents its release discipline, read it and check against that rather than against generic assumptions.
