# alexandria

> 「その仕様、CLAUDE.md に書いといて」
> 「README にも書いておきますね」
> 「docs/ にも同じ説明があった気がする」
> 「三箇所とも、微妙に違うことが書いてありますが」
> 「……どれが本当？」

三冊の本に三通りの真実が書いてある図書館を、人は図書館と呼びません。呼ぶとしたら、物置です。`alexandria` は、その物置を図書館に変えるプラグインです。

名前は、かつて世界のすべての知識に住所を与えようとした大図書館から取りました。あの館の教訓は二つあります。知識は、集めただけでは残らないということ。そして、燃えてから悔やんでも遅いということです。

## これは何？

プロジェクトの書かれた知識を、それぞれ性質の違う四つの棚と一枚の案内板に配架し、棚に並んだ本の中身がいまも真実かを点検する図書館です。

- **CLAUDE.md** は館の憲章です。価値観と判断基準だけを書きます。コードベースを明日ゼロから書き直しても生き残る文だけが、ここに住む資格を持ちます
- **docs/** は現在の蔵書です。「今この瞬間に何があるか」だけを書きます。構成、手順、データ契約。読むのはコードを書く前で、書いた後の言い訳のためではありません
- **adr/** は決定の書庫です。「なぜそう決めたか、何を捨てたか」を時系列で綴じます。一度採用された記録は書き換えず、判断が変わったら新しい一枚で上書きではなく置き換えます。すべての決定が不変の記録として時系列に刻まれていく様は、世界の全記憶が記されるというアカシックレコードに少し似ています。ただしこちらは実在して、grep が効きます。書庫はルート直下に建ちます。決定の記録は現状説明の押し入れに仕舞う物ではないからです
- **references/** は標本室です。スプレッドシートのサンプル行、捕獲したAPIレスポンス。散文にせず、加工せず、コードからも読み込まない生データの棚です
- **README.md** は玄関の案内板です。初めて来た人に、ここが何の館で、どの棚に何があるかを示します

配架の原則はただ一つ、**一つの事実に、住所は一つ**。同じ内容が二つの棚にあると、現実が動いたときにどちらか一方だけが更新され、読者はもうどちらも信じられなくなります。腐った文書は、信頼されているぶん、無い文書より悪いのです。

## 特徴

- **配架には必ず証拠を添えます**: 「このファイルのこの節は、書き直し後も生き残る価値観ではなく今日の手順なので、docs/ 行きです」。ファイル名の雰囲気で本を動かす司書は、司書ではなく引っ越し業者です
- **蔵書点検が中身の嘘を暴きます**: 完璧に整列した棚に、三ヶ月前の真実が並んでいることはよくあります。点検は各文書の主張をディスク上の現実と突き合わせ、「ページは X と言うが、リポジトリは Y を示す」の形で報告します。採用後にこっそり書き換えられた ADR は、git の履歴が覚えています
- **診断と整理は別の動作です**: 監査も点検も、まず報告を差し出し、あなたが頷いた項目だけを直します。移動は `git mv` で履歴ごと運び、リンクと索引も同じ一手で追随させます。リンク切れを残す移動は、移動ではなく遺失です
- **綺麗な館には綺麗と言います**: 違反ゼロの監査報告も、全書合格の点検報告も、空振りではなく成果です。仕事を見せるために問題を発明したりはしません
- **空の棚は作りません**: 生データの無いプロジェクトに references/ は建てません。空の標本室は家具ではなく、ただの埃です

## コンポーネント

| 種別 | 名前 | 説明 |
|------|------|------|
| Command | `/alexandria:audit` | 全文書を四層に写像し、配架違反を証拠つきで報告。承認された配架案だけを実行する |
| Command | `/alexandria:inspect` | 蔵書点検。各文書の主張を現実と突き合わせ、古びた記述・切れたリンク・書き換えられた記録を報告する |
| Command | `/alexandria:setup` | 文書構造の無いプロジェクトに、四層の棚と案内板を実際の観察に基づいて敷設する |
| Skill | `four-track-contract` | 四層知識契約の教義・配架手順・点検手順・雛形。「この情報どこに書く?」に随時答える |

## 使い方

散らかった館には監査を。

```
/alexandria:audit
```

本の中身が信じられなくなったら点検を。

```
/alexandria:inspect
```

更地には敷設を。

```
/alexandria:setup
```

会話の途中で「これって docs/ と README どっちに書くべき?」と聞くだけでも、スキルが教義から答えます。コマンドの儀式は要りません。

## setup が最初にやる、少し行儀のいいこと

敷設のとき、館は最初の二冊の決定記録を自分で書きます。「ADR を使うという決定」と「四層契約を採用するという決定」です。日付は敷設した当日、却下した代替案も正直に記します。「決定を記録せよ」と説く構造が自分の誕生の記録を持っていなかったら、その館は開館初日から自己矛盾で軋むからです。

## よくある質問

**Q: 全部 CLAUDE.md に書いてはだめですか？**
A: だめではありません。ただし価値観と現状と判断理由を一冊に綴じると、現状が変わるたびに理由が消えるか、理由を守って現状の記述が腐るか、必ずどちらかが起きます。三年もののCLAUDE.mdの中で、今も真実の段落がどれか当てるゲームをしたことがある人は、この分割の理由をすでに体で知っています。

**Q: adr/ がルートにあるのはなぜですか。docs/adr/ のほうが片付いて見えます。**
A: 片付いて見えるのは、決定の記録を現状説明の子分として扱っているからです。「今どうなっているか」と「なぜそうしたか」は親子ではなく、対等な別トラックです。書庫を書斎の押し入れに入れる図書館はありません。

**Q: 配架が正しければ、中身も正しいですよね？**
A: 別の問題です。全部の本が正しい棚にあって、その全部が三年前の現実を語っている館は普通に存在します。だから棚の監査と蔵書の点検は別のコマンドです。棚は audit、中身は inspect が受け持ちます。

**Q: 勝手にファイルを動かしませんか？**
A: 動かしません。館は配架案と点検報告をカウンターに置き、あなたが頷いた分だけ手を入れます。二つの棚に等しく収まりうる本は、勝手に分類せず、あなたに尋ねます。本の意味を決めるのは司書の仕事ではないからです。

**Q: 一度整理したら、もう散らからないですよね？**
A: 散らかります。図書館は、開館している限り散らかり続けます。だから司書は常勤で雇うのです。大きな変更のあとに、また audit と inspect を呼んでください。

**Q: 私のプロジェクトは小さくて、四層も要らない気がします。**
A: 正しい感覚です。この契約は「知識があるならここに住む」を定めるもので、棚を埋めるために知識を製造しろとは言いません。二枚のファイルで足りる館には、二枚のファイルと、それがどこにあるかの案内板だけで十分です。

**Q: アレクサンドリアって、最後は燃えた館の名前では？**
A: だからこの名前です。あの館は、知識を集めることと残すことを混同しました。このプラグインが受け持つのは配架と点検、つまり残す側の仕事です。二度と燃やさないための名前だと思ってください。

## できないこと・正直な制約

- **点検は照合であって、審査ではありません。** 文章の巧拙や設計の良し悪しは判定しません。間違った推論を正確に記録した ADR は、点検の目には健全な記録です
- **照合できない主張は「照合できない」と報告します。** 実機やリモートの状態を語る文書の真偽は、リポジトリの中からは確かめられません。references/ の標本と実機のずれも、コード側の期待と食い違ったときに間接的に気づけるだけです
- **古びた文書の書き直しは提案までです。** 死んだリンクや改名されたパスは承認のうえ直しますが、消えたアーキテクチャを語るページの再執筆は、書ける知識を持つ人の仕事として報告します。発見を閉じるために中身を発明したら、古い真実が新しい虚構に変わるだけだからです
- **分類は最終的に人間の判断です。** 価値観と現状の境界には、本当に曖昧な本が存在します。館は基準と推薦を示しますが、票は持ちません
- **静的サイト生成ツールが所有する docs/ には手を出しません。** mkdocs.yml や docusaurus.config が棚の構造を決めている館では、発見は報告に留め、動かすかどうかはあなたが決めます
